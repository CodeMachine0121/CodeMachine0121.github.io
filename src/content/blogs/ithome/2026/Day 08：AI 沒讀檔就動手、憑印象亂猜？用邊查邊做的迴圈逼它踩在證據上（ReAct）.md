## 從一次自信滿滿的瞎猜說起

昨天的 CoT 解決的是「純邏輯題跳步算錯」——把推理攤開再下 code。但它有個天生的邊界：那套推理全在 agent 腦子裡跑，不碰外面的世界。題目自帶所有條件時，這很好用；一旦任務要去讀專案裡的檔案、要打某個外部服務，光靠腦內推理就會出事。

你八成也撞過這種場面。Day 04 你把 fetch-and-summarize 封裝成 skill，拿幾個乖乖的來源測都沒問題。現在你想讓稍後閱讀服務一次支援一批新來源——各種部落格、新聞站、電子報——於是請 agent 把這批來源的抓取都補進 `internal/fetcher`。它想都沒想就開始寫：用 `net/http` 發個 GET、拿 goquery 撈 `article` 標籤、假設回來的都是乾淨 HTML，接著把 parser 一路寫完，還很有信心地告訴你「這樣就能處理大部分網站」。

問題是它從頭到尾沒去看過任何一個目標網站。它不知道要抓的來源裡，有一半提供 RSS（根本不用爬 HTML）；也不知道其中幾個是前端渲染的 SPA，直接抓 HTML 只會拿到一個空殼，得上 headless 瀏覽器才行。它寫的東西在紙面上很完整，一放到真實來源上就整片崩掉。

這不是它笨，是它被丟進資訊真空裡動手。它把「印象中網頁大概長這樣」當成了事實。今天要補的，就是 CoT 缺的那一塊：讓推理跟真實環境形成回饋迴路，每動一步之前，先去看一眼。

## 這其實就是 agent 最小的那個內迴圈

在講怎麼用之前，先把它放回系列的主軸。一個 agent 的本質，就是一個 act → observe → verify → repeat 的迴圈；而今天這套 Thought → Action → Observation，正是這個迴圈縮到最小的樣子——想一下要幹嘛（act 的前置）、動手做一個實際動作（act）、把真實回傳收進來（observe），然後回到下一輪。它靠 observe 這一步，讓推理踩在真實回饋上，而不是踩在模型腦補的世界上。

這件事之所以重要，是因為本系列後面所有更大的迴圈——生產者對審查者、辯論對裁判、紅綠測試、回歸驗證——全都建立在這個內迴圈上。你先在這裡把「動一步、看一眼、再決定下一步」這個最小動作練熟，後面把它擴成多角色、擴成自動化，才有地基。所以今天與其說是在學一個叫 ReAct 的招式，不如說是在建整個 Loop Engineering 最裡面那一圈。招式名字（ReAct）記不記得都行，這個「observe 讓推理落地」的動作才是主角。

## 判斷：什麼時候該套這個迴圈

先講清楚什麼場景值得動用這套迴圈，免得小題大作。

判準其實只有一條：**下一步該怎麼做，取決於上一步做完才看得到的東西嗎？** 如果答案是「要，得先看過才知道」，那就是這個迴圈的場子。抓網頁內容就是這樣——你得先探這個來源有沒有 RSS，答案會決定接下來是走 feed parser 還是爬 HTML；爬回來是空殼還是有內容，又決定要不要上 headless。每一步的決策都依賴前一步的觀察，沒有人能一開始就把整條路想死。

反過來，什麼時候是小題大作？當任務是**封閉的純計算**——所有條件題目都給了，不需要去外面查任何東西。昨天那個週報 `dedupeAndRank` 的去重、排序邏輯就是這樣：輸入是一組已知的文章、規則是你定的，agent 在腦中把推理攤開就能寫對，硬要它「先 Action 一下」反而是多餘的來回。這種題用 CoT 就夠。

一個好記的分法：**CoT 只用它腦中已有的資訊，這個迴圈則會去外面查。** 前者靠把腦裡的東西整理清楚就能作答；後者的重點是知道該去查什麼、查完再決定下一步。凡是要跟檔案系統、API、外部工具互動，而且路徑會隨觀察而變的，就屬於後者。

## 建立：Thought → Action → Observation 的迴圈

這個內迴圈的骨架是三個動作反覆交錯：

- **Thought（想）**：根據目前已知的資訊，判斷「接下來我需要知道什麼／該做什麼」。
- **Action（做）**：執行一個實際動作去取得資訊或改變狀態——讀一個檔、發一個請求、跑一段指令。
- **Observation（看）**：把動作真實的回傳結果收進來，當成下一輪 Thought 的依據。

關鍵在最後那個 Observation。CoT 的鏈條是「想 → 想 → 想」，一路在腦中推下去，沒有東西能把錯誤的假設打回來；這個迴圈在每個「想」之間都插進一次真實世界的回話。假設錯了，Observation 會當場打臉，逼下一個 Thought 修正方向。抗幻覺的能力就是從這裡來的——它不再需要「猜對」，只需要「去看」。

但光有這個骨架不夠，放任 agent 自己跑，它常會犯兩種毛病：一是跑著跑著又飄回瞎猜，明明沒去看就宣稱「這個網站應該有 RSS」；二是陷進無止境的探查，一直查一直查，遲遲不肯收手動工。所以我們要用 rule 把這個迴圈的紀律編譯進去，把三件事變成硬性協議：

1. **每個 Thought 必須引用上一個 Observation。** 不能憑空推進，你的下一步得建立在剛剛真的看到的東西上。
2. **設步數上限。** 給探查一個明確的次數（例如最多五輪），逼它別無限打轉。
3. **證據夠了才 Finish。** 收手的條件要明講：關鍵問題都有觀察支撐了，才能進入實作，不能靠感覺喊停。

## 用 Claude Code 落地

把上面三條寫成一份 rule，放進 `.claude/rules/react-protocol.md`。它不會憑空給 agent 新能力，但會把「邊查邊做」的紀律固定下來，讓每次碰環境的任務都照同一套走：

```markdown
# ReAct 協議：碰環境的任務先探再做

適用範圍：任何需要讀專案檔案、打外部 API、或動作結果會決定下一步的任務。

## 執行迴圈
以 Thought → Action → Observation 交錯進行，直到 Finish：

- Thought：說明「基於上一個 Observation，我現在需要知道什麼」。
  第一個 Thought 以外，每個 Thought MUST 明確引用上一個 Observation 的內容。
- Action：執行一個具體動作（讀檔、發請求、跑指令），一次只做一件。
- Observation：貼上動作的真實回傳，不得杜撰或臆測結果。

## 紀律
- 步數上限 5 輪。到上限仍無法收斂，停下並回報卡在哪。
- NEVER 在沒有對應 Observation 的情況下宣稱事實（例如「這網站有 RSS」）。
- Finish 條件：影響方案選擇的關鍵問題都已有 Observation 支撐，才進入實作。
- 若某個 Observation 推翻了先前假設，MUST 在下一個 Thought 明講「假設被推翻，改走 X」。
```

有了這份協議，同一個「抓網頁內容」的任務，agent 跑出來的軌跡會變成這樣：

```
Thought 1: 要抓這個來源的內容，最省事的是有 RSS。先確認有沒有 feed。
Action 1:  GET https://example-blog.com/feed
Observation 1: 200 OK，回傳合法的 RSS XML，含 <item> 與 <content:encoded>。

Thought 2: Observation 1 顯示有 RSS 且帶全文，那就不必爬 HTML。
           但使用者存的另一個來源是 news-spa.com，得分開確認。
Action 2:  GET https://news-spa.com/article/123
Observation 2: 200 OK，但 body 幾乎是空的 <div id="root">，內容靠 JS 載入。

Thought 3: Observation 2 推翻了「直接抓 HTML 就有內容」的假設——
           news-spa.com 是前端渲染，直抓只拿到空殼，改走 headless。
Action 3:  用 chromedp 載入同一頁，等 networkidle 後取 DOM
Observation 3: 取得完整文章正文，約 4200 字。

Thought 4: 兩類來源都驗證過了：有 RSS 走 feed，SPA 走 headless，
           其餘走一般 HTML 擷取。關鍵問題都有觀察支撐，可以動工。
Finish:    fetcher 設計成三分支——RSS / headless / 靜態 HTML，依探測結果分流。
```

注意 Observation 2 那一步：它當場推翻了 agent 原本「直接抓 HTML 就好」的假設，Thought 3 也照協議明講了「假設被推翻，改走 headless」。這正是昨天那次瞎猜缺的東西——它終於是踩在看到的結果上做決定，而不是踩在想像上。實作分流時，靜態 HTML 用 `net/http` 加 goquery、SPA 走 chromedp、RSS 走 feed 解析，三條分支各自對應探測出來的來源型別。

## 對照概念：推理與行動的交錯

把場景拉高一層，這對應到 agent 設計裡一個被反覆討論的模式：**把「推理」和「工具使用」交錯成一個循環**，而不是先把整套計畫想完再一次執行（這個模式業界叫 ReAct）。

昨天的 CoT 是純推理，鏈條從頭到尾在模型內部展開，不與外界互動。它的弱點也正在這：模型只能拿訓練時見過的世界當依據，遇到「這個特定專案、這個特定網站現在長怎樣」這類它無從得知的事實，就只能編。今天這個模式的核心貢獻，是在推理鏈裡插進「行動—觀察」這一環，讓每一步推理都有機會被真實回傳校正。差別其實就一句：**有沒有跟環境形成回饋迴路。** CoT 沒有，這個迴圈有。

也因為有這個迴路，它在工程上特別實用——絕大多數真實開發任務都不是封閉題，而是「得先看看現在的 code／資料／服務長怎樣」才能往下走的開放題。這也是為什麼它值得當整個系列的地基：後面每個更大的迴圈，本質都是在這個「動一步、看一眼、再決定」的內迴圈外面再包一層。

## 今日實踐任務

今天用這個內迴圈把稍後閱讀服務這批新來源的抓取補起來，重點不在寫多快，而在讓每一步都取決於上一步的觀察。

1. 把上面的 react-protocol rule 放進 `.claude/rules/`，確認碰環境的任務會套用到它。
2. 請 agent 為現有的幾個來源設計 `internal/fetcher` 的抓取方案，要求它照 Thought → Action → Observation 跑：先探每個來源有沒有 RSS、抓回來是不是空殼、要不要上 headless，再決定分幾條分支。實作端對齊 Go——靜態 HTML 用 `net/http` 加 goquery、SPA 用 chromedp，並用 `go test` 針對三種分支各補一個表格驅動測試。
3. 全程盯著它的軌跡，別讓它跳過 Observation 直接下結論。

驗收標準：

- 完整軌跡裡至少出現一次「Observation 推翻先前假設、下一個 Thought 明講改變方向」——這是它真的在看、而非在猜的證據。
- 每個 Thought 都掛得上前一個 Observation，沒有憑空冒出的事實宣稱。
- 有命中停止條件：不是無限探查，而是在關鍵問題都有觀察支撐後就 Finish 動工，或到步數上限時停下回報卡點。

## 明天 Day 09

今天這個內迴圈是一條路走到底、邊走邊修——它假設你大方向對了，只要靠觀察把細節校準就好。但有些問題不是這樣：一段爛 code 有好幾種救法、一個功能有好幾種設計走向，你事先根本不知道哪條路會通，得攤開來試錯比較。明天換一個推理形狀——不再是單線推進，而是同時展開多條路徑、各自評估後選優、淘汰劣枝再回溯（Tree-of-Thoughts）。
