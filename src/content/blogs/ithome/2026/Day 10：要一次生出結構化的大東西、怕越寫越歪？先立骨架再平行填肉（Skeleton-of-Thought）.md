## 越寫越歪的那個下午

過去這三天，我們一路在談 agent「怎麼想」。Chain-of-Thought 是讓它把步驟攤開來想；ReAct 是想一步、動一步、再看結果修正；昨天的 Tree-of-Thoughts 則是同時展開好幾條路徑，最後選一條最好的走。三種都在解「思考」的問題。但有一種痛，跟「怎麼想」關係不大，跟「怎麼一次產出一個結構化的大東西」關係很大。

我在稍後閱讀工具上想加一整套前端頁面：文章列表頁、單篇詳情頁、每週週報頁、設定頁。我很自然地丟了一句「幫我把這幾頁的 UI 生出來」。agent 開始寫，列表頁寫得挺好，詳情頁也還行。可是寫到週報頁的時候，我發現它引用的資料欄位跟列表頁對不上；設定頁的按鈕樣式又自己發明了一套。等它全部寫完，我拿到的是幾頁「各自都能看、湊在一起像好幾個人做的」東西。

這跟人熬夜趕一份長文件很像：寫到第五章，你其實已經不太記得第一章用了哪些詞，於是同一個概念換了三種叫法。

這不是它笨。這是「一邊生成一邊決定結構」這種寫法本身的毛病。今天的 Skeleton-of-Thought 就是拆這個毛病：先立骨架，確認結構，再平行填肉。

## 為什麼線性生成大東西會越寫越歪

模型在生成長內容時，是一個 token 接一個 token 往下續的。它每寫一段，能參考的就是「已經寫出來的前文，加上你的指令」。麻煩在於，很多結構性的決策——整體要分幾塊、每塊負責什麼、彼此怎麼呼應、命名規則是什麼——是在它寫到第一段的時候就順手定下的，而且從來沒被明確寫出來過，只是隱含在它已生成的文字裡。

於是兩件事開始累積誤差。第一，結構是「邊寫邊長出來的」，不是先想好的。它寫到後面才發現前面的切法不太對，但已經回不去了，只能將錯就錯往下接。第二，越往後，它要同時兼顧的前文越長，一致性的維護成本越高，注意力被稀釋，前面立的規矩就慢慢守不住。

關鍵的體悟是：一致性最好的保證，是讓所有部分共用同一份「已經確定、而且明確寫出來的」結構，而不是指望模型在漫長的線性生成裡自己記著。既然如此，那就把結構這件事，從內容生成裡抽出來、先單獨做完。

## 判斷：什麼時候該用、什麼時候不必

三個訊號同時成立才划算：產出「大」、結構性「強」、各段可「獨立填」。判斷那把尺很樸素：這東西大到我怕它寫到後面對不上自己嗎？

反過來，如果你要的東西又短又單一，比方說「幫這個 parser 加個錯誤處理」，那就直接寫，硬要它先產一份骨架只是多繞一圈。還有一種反例要特別小心：如果一段內容的寫法會強烈牽動另一段（例如演算法前半段的資料結構決定後半段怎麼寫），那本質上就不是「可獨立填」，硬拆平行反而出事。

## 建立：先產骨架，確認後再填肉

方法本身只有三步：

第一步，只要骨架，不要內容。讓它先交出整體結構——分成哪幾塊、每塊負責什麼、共用哪些東西、命名怎麼定，先不要碰任何實作。

第二步，也是最重要的一步：你先確認並鎖定骨架，再讓它往下走。骨架階段的錯改起來便宜，等肉都長滿再改結構，等於整份重寫。

第三步，骨架定案後各段平行填。因為結構已經是共識，分開生成也不會漂走，順帶把速度賺回來。

顯性化正是它比「一次寫完」強的地方：過去「先想結構」這件事你在腦子裡做，現在你要求 agent 把「骨架」這個中間產物明確交出來，讓它能被檢查、確認，並當成後續平行工作的共同依據。

## 用 Claude Code 落地

### 第一步：讓它只產骨架

```
你是這個 side project（稍後閱讀 + AI 週報摘要工具）的前端架構者。
現在只做一件事：產出多頁 UI 的骨架，不要寫任何實作程式碼。

請用以下格式，為每一頁輸出一個節點：

## 頁面：<名稱>
- 路由：
- 目的：一句話
- 使用的共用元件：（只能從下方「共用元件清單」挑）
- 區塊結構：由上到下列出區塊（不含樣式細節）
- 驗收條件：3 條，可被檢查

最後另外輸出一份全域約束：
- 共用元件清單（所有頁面共用，名稱一次定死）
- 命名規則（元件、CSS class、路由）
- 視覺基調（僅一句）

頁面範圍：登入、文章列表、單篇閱讀、週報、設定。
現在只輸出骨架與全域約束，不要寫任何 UI 程式碼。
```

### 第二步：你確認並鎖定骨架

拿到骨架，花兩分鐘：檢查共用元件清單有沒有漏；檢查五頁的區塊結構彼此對稱；把改好的骨架貼回去，明確說一句：骨架就定這樣，接下來所有頁面 MUST 遵守這份約束，不得新增或改名共用元件。

這一步千萬別省。第二步沒真的鎖定，後面平行填肉就等於各寫各的。

### 第三步：平行填肉（用 subagent 分派）

```
骨架已鎖定（如上）。現在平行填肉。
請為以下每一頁各派一個 subagent，同時進行：登入、文章列表、單篇閱讀、週報、設定。

每個 subagent 的輸入固定為兩塊：
1. 全域約束（共用元件清單 / 命名規則 / 視覺基調）——原文照抄，不得改動。
2. 該頁的骨架節點（區塊結構 + 驗收條件）。

每個 subagent 的產出要求：
- 只實作自己那一頁，其他頁不要碰。
- 共用元件一律 import，NEVER 自己重寫或改名。
- 交件前自己對照該頁的 3 條驗收條件逐條檢查，並在結尾附一份自查結果。

全部回來後，主流程再做一次跨頁一致性檢查。
```

### 收尾：跨段整合檢查

平行的代價是各段可能有細微偏差，最後一定要補一次整合檢查：把五頁擺在一起，只看跨頁一致性——共用元件是不是真的共用、命名有沒有分岔、視覺基調有沒有跑掉。這一步交給主流程做，不要讓任何填肉的 subagent 去評判全局。

## 對照概念：這跟「先規劃再執行」「平行化」差在哪

它跟「先規劃再執行」共享同一個直覺：別急著動手，先把結構想清楚。差別在目的。一般的先規劃再執行，規劃是為了排出正確的執行順序，重點在流程對不對；計畫做完常常就丟了。Skeleton-of-Thought 的骨架不是為了排順序，而是為了產出一份會被所有後續生成共用的結構契約，重點在一致性與可平行——這份骨架是要一路被引用到最後的活文件。

它跟「平行化」也不同。單純的平行化是把彼此無關的任務同時跑，目的是省時間，各任務之間沒有共同前提。Skeleton-of-Thought 的平行是「受約束的平行」——每一段都被同一份骨架綁住，平行是結果不是目的。它真正買到的不是速度，是一致性。

## 今日實踐任務

1. 線性組：開一個乾淨對話，直接要 agent 一次生出五頁 UI，不給骨架。存下結果。
2. 骨架組：用第一步的骨架 spec 產出骨架，自己確認並鎖定，再平行填肉。存下結果。
3. 對照兩組一致性，逐項打勾：共用元件是否真的共用；命名是否統一；視覺基調是否一致；每頁是否滿足自己骨架的 3 條驗收條件。

驗收標準：骨架組在「共用元件版本數」與「命名分岔數」上明顯少於線性組，且五頁都能對上各自的驗收條件。若沒有明顯較好，多半是骨架不夠細，或第二步沒真的鎖定。
