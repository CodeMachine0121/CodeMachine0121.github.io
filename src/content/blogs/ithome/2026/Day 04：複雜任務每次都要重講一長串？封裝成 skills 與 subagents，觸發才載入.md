## 又要把那段話重打一次

昨天談 rules，結論不太樂觀：它是行為準則，但沒有強制力——會被讀，不保證被照做。今天換個角度。規矩之外，還有一類東西也很煩：**你每次都要重講一遍的複雜任務**。

回到那個「稍後閱讀 + AI 週報摘要」的小工具，最先卡住的往往就是「抓一個網頁連結、把內文抽出來、產一段摘要」這件事。聽起來很單純，實際對 AI 交代起來卻是一長串：

> 幫我抓這個 URL 的內容。注意先看看有沒有 RSS 或 API，有的話優先用；如果是那種整頁 JS 才渲染的站，要用 headless 的方式；抓下來要先把導覽列、廣告、頁尾這些雜訊去掉，只留正文；然後摘要控制在三段以內，第一段講它在談什麼、第二段講重點、第三段講對誰有用；輸出用 Markdown，前面加上原文標題跟連結……

第一次講完，結果還行。問題是隔天開新對話，你又得把這段從頭打一次。重講到第三次，你大概就會發現不對——這種每次都要重述的長前提，本身就是一種待處理的技術債。它不該活在你的短期記憶裡，該被固定下來。

換個角度看：你在建的那個「想→動→看→修」開發迴圈，每一輪都得先重建一次前提，這迴圈就轉不順。把重複的複雜任務封裝起來，等於幫迴圈裝上一個現成零件——需要時觸發、跑完就退場，迴圈不必每輪都從零重講一遍。今天要建的 skill 與 subagent，就是這個零件。

## 判斷：這任務該封裝嗎？該封裝成什麼？

不是每件事都值得封裝。動手前先過兩道判斷。

第一道：**這任務會重複出現嗎？前提夠不夠長？** 一次性的雜事直接講就好，封裝反而是多餘的儀式。但如果它同時滿足「常常要做」「每次都得交代一串前提或步驟」「步驟之間有固定順序」，那它就是好的封裝對象。「抓網頁產摘要」三條全中。

第二道，也是今天的重點：**該封裝成 skill，還是開一個 subagent？** 兩者都是「觸發才載入的能力」，但定位不同：

|         | skill            | subagent（分身）               |
|---------|------------------|----------------------------|
| 本質      | 一包被觸發時載入的指令＋流程   | 一個有獨立 context 的子任務執行者      |
| context | 和主對話共用同一份        | 自己開一份乾淨的，跑完只回結果            |
| 適合      | 「照這套步驟做這件事」的固定流程 | 需要大量中間探索、翻很多檔、產出一堆過程雜訊的重任務 |
| 產出      | 直接在當前對話裡完成       | 把過程留在自己的 context，只把結論帶回主對話 |

一句話分辨：**如果你在意的是「把步驟講清楚、讓它照做」，用 skill；如果你在意的是「這任務會塞爆主對話的 context，我只想要結果」，用 subagent。**

抓網頁這件事有意思，它其實兩種都用得上。「怎麼抽正文、摘要寫幾段」是固定流程，適合寫成 skill；但真的去抓一個站時，agent 可能要試 RSS、試 net/http 直接請求、試 chromedp headless，中間會產生一大堆試錯的觀察——這些過程你多半完全不想看到，只想要最後那段乾淨摘要。這時候讓 subagent 去跑，主對話就清爽了。

## 「觸發才載入」為什麼重要

這是 skill 和 subagent 共同的關鍵性質，也是它們比「寫進 CLAUDE.md」更值得的原因。

你當然可以把那段抓網頁的長前提塞進 CLAUDE.md，這樣每次開場就會載入。但 CLAUDE.md 是**每次都載入**——不管這次對話要不要抓網頁，那一大段指令都佔著 context。你的 CLAUDE.md 會愈長愈臃腫，真正的專案慣例反而被淹沒。

skill 和 subagent 走的是另一條路：**平常只留一行簡短的觸發描述，被判斷用得上時，才把完整內容載進來。** 沒要抓網頁的那些對話，這包東西就靜靜躺著，不佔 context 也不干擾。這件事很像「按需檢索」——不是把所有知識一次攤開，而是要用的時候才去取。省下來的 context，就是你留給真正手上任務的空間。

## 建立：把重複的複雜任務封裝起來

封裝的原則很簡單：**把你每次口述的那一長串，寫成檔案。** 一個 skill 大致就三個部分：

1. **觸發描述**：一句話講「什麼情況下該用這個 skill」，這是平常唯一被載入的部分，agent 靠它判斷要不要展開。寫得夠準，才不會該觸發沒觸發、或不該觸發亂觸發。
2. **步驟與前提**：把你原本口述的流程攤成清楚的步驟，包含那些容易漏掉的邊界條件（優先找 RSS、JS 站用 headless、去雜訊）。
3. **輸出格式**：明確規定產出長什麼樣（幾段、Markdown、要不要標題連結），這樣每次結果才一致。

subagent 的定義也類似，額外多一件事：因為它跑在獨立 context，你要交代清楚「它拿得到什麼輸入、該回傳什麼結果」——它不會看到你主對話裡的來龍去脈，只看得到你交給它的東西。

## 用 Claude Code 落地

在 Claude Code 裡，skill 放在 `.claude/skills/` 底下，一個資料夾一個 skill，核心是一份帶 frontmatter 的說明檔。抓網頁摘要的 skill 大概長這樣：

```markdown
---
name: fetch-and-summarize
description: 當使用者提供一個文章 URL、要求抓內容並產出摘要時使用。負責取得正文、去雜訊、產出三段式 Markdown 摘要。
---

# 抓網頁並產摘要

抓取與正文清理的邏輯放在 `internal/fetcher`，摘要放在 `internal/summary`；改動後跑 `go test ./internal/fetcher/... ./internal/summary/...` 確認。

## 步驟
1. 先偵測來源型別：優先找 RSS / API；純靜態頁直接用 net/http 請求；JS 才渲染的站改用 headless（chromedp）。
2. 抽出正文，去掉導覽列、廣告、頁尾等雜訊，只留內文（HTML 解析用 goquery）。
3. 產出摘要，控制在三段以內：
   - 第一段：這篇在談什麼
   - 第二段：核心重點
   - 第三段：對誰有用
4. 若正文抽取失敗，回傳 error 說明失敗原因，不要硬編一段摘要。

## 輸出格式
- 開頭一行：原文標題 + 連結
- 接著三段式摘要，純 Markdown
```

那個 `description` 就是「觸發描述」——平常只有它被載入，agent 判斷這次任務對得上，才把整份步驟展開。之後不管哪個對話，你只要說「幫我用 fetch-and-summarize 處理這個連結」，或甚至只是丟一個 URL 說要摘要，它就會自己載入這套流程，你再也不用重打那一長串。

subagent 則放在 `.claude/agents/`，定義上多了「它是誰、能用哪些工具」：

```markdown
---
name: web-summarizer
description: 獨立跑抓網頁與摘要的重任務，過程留在自己的 context，只把最終摘要帶回主對話。
tools: Read, Bash, WebFetch
---

你是專門處理「抓網頁 → 產摘要」的分身，跑在這個 Go 後端專案裡。
給你一個 URL，你負責完成抓取與摘要：可讀 internal/fetcher、internal/summary 的既有實作，
需要驗證時用 Bash 跑 go test。中間試 RSS、試 net/http、試 chromedp 的試錯與雜訊留在你自己這裡，
最後只回傳一段乾淨的三段式 Markdown 摘要給呼叫你的人。
```

差別就在這個「獨立 context」：subagent 去試 RSS、試 net/http、試 chromedp 那些過程，都留在它自己那份 context 裡，主對話只收到最後那段摘要。要處理一大批連結時，這個隔離很值錢——主對話不會被幾十次抓取的日誌塞爆。

## 對照概念：工具使用與分身

把鏡頭拉遠一點，這兩件事各自對應到 agent 領域裡談了很久的概念。

skill 呼叫外部能力、照固定流程完成一件事，對應的是**工具使用（function calling）**——讓模型不只會講話，還能觸發定義好的動作，並把結果接回推理裡。skill 是這個概念很務實的一種封裝：把「一組動作＋怎麼用它們的流程」打包成一個可重複觸發的單位。

subagent 對應的則是**多代理協作**裡的「分身」概念——當單一 agent 的 context 顧不過來，就把某類子任務外包給一個獨立的執行者，它自己想、自己做，只把結果交回來。今天我們用的是最輕量的形式：主對話當協調者，一個分身負責一種重任務。這條線會一路延伸到 Part 2，到時候會用多個分身組出更完整的產線。

回到迴圈的角度：skill 正是迴圈的「Skills」零件——把重複的複雜任務封裝起來，消除每輪都要重講前提的那筆 intent debt，讓迴圈不必每次重建前提就能往下轉（工具使用與多代理協作是它們背後的學名，記不記得都不影響用）。

不用記這些學名也沒關係。實務上你只要抓住那個判斷：**固定流程、想在當前對話完成 → skill；context 會爆、只要結果 → subagent。**

## 今日實踐任務

把範例專案的「抓網頁內容並產摘要」正式封裝起來，讓它變成可重複觸發的能力：

1. 在專案的 `.claude/skills/` 下建立 `fetch-and-summarize`，把你原本口述的抓取與摘要流程寫成上面那樣的說明檔，`description` 要能準確描述「什麼時候該用它」。
2. 如果你預期之後會一次處理很多連結，順手在 `.claude/agents/` 建一個 `web-summarizer` subagent，交代清楚輸入與回傳。
3. 開一個全新對話（不要在剛才那個），只丟一個文章 URL、說你要摘要，看它會不會自己載入這套流程。

驗收標準：

- 新對話裡你完全沒有重述那一長串前提，它就照你封裝的步驟產出了三段式摘要。
- 產出格式和你在 skill 裡規定的一致（標題連結 + 三段）。
- 沒要摘要的對話裡，你感覺不到這包東西的存在——它沒有平白佔用 context。

做到這裡，你等於把一段每天重打的技術債，一次性還清了。
