## 從一個令人洩氣的早上說起

昨天我們把 hooks 和 permissions 都設好了，enforce 這一層總算補齊。context（會讀）、invoke（觸發才載入）、enforce（真正攔得下）三層機制，紙面上看起來很完整。但在真正把它們湊成一套之前，有個常見的誤解得先澄清，不然你會白忙一場。這篇要做的，其實是替迴圈的地基收尾——把前幾天散落的零件（State/Memory、Skills、Guardrails）按載入時機歸位，收斂成一個可以整包 clone 的 starter，讓「想→動→看→修」這個內迴圈之後每次都踩在同一塊地基上。

情境是這樣。假設你在稍後閱讀工具的 repo 裡開了一個 `.claude/docs/` 目錄，把摘要模組的設計筆記、資料庫 schema、還有幾份 API 慣例都寫了進去。心裡想的是：這下 agent 開發時該有的背景都齊了，它自己會去參考。

結果隔天早上，你請它「照我們的摘要格式慣例，幫這個新來源加一個 parser」。它產出的東西，格式跟你在 `.claude/docs/summary-format.md` 裡寫的完全對不上。你一度以為是它不聽話，翻了半天才會意識到——它根本沒讀到那份檔案。檔案好端端躺在目錄裡，但從沒進過這次對話的 context。

這是很多人第一次整理 `.claude/` 時都會踩的坑：**把檔案放進目錄，跟那份檔案進入 AI 的 context，是兩件完全不同的事。**

## 判斷：什麼時候檔案才會被讀到

要避開這個坑，得先弄清楚 agent 到底怎麼決定「這次對話要看哪些檔案」。大致分三種情況，對得上前幾天講的三層。

第一種是**啟動時自動載入**。這一類數量很少，而且是硬規則。專案根目錄的 CLAUDE.md（Day 02 談的長期記憶）、user 層的 memory，這些在每次開新對話時就會被讀進 context，你不用做任何事。正因為它自動載入、每次都占 context，所以只該放「每次都用得到」的專案慣例，不該把整份設計文件塞進去。

第二種是**觸發才載入**。Day 04 的 skills、subagents 屬於這一類。它們平常不占 context，只有在符合觸發條件（例如任務描述對上了 skill 的用途）時才被拉進來。這是為了省 context——把不常用的能力收起來，需要時才展開。

第三種，也是最容易被誤會的，是**明確引用才載入**。放在 `.claude/docs/` 底下的設計筆記、schema、慣例文件，絕大多數屬於這一類。它們不會自動進 context，也沒有觸發機制；只有當你在對話裡明確叫 agent 去讀（「先看 `.claude/docs/summary-format.md` 再動手」），或它自己判斷需要而主動去讀時，內容才真的進來。

前面那個早上的錯，就是把第三種當成了第一種。你以為「放進 `.claude/` 就等於自動載入」，但 `.claude/docs/` 對 agent 來說更像一座圖書館——書都在架上，可是沒人替你翻開。

所以 context 邊界的判斷，其實可以濃縮成一個問題：**這份檔案是「每次都要對得上的鐵則」，還是「需要時去查的參考資料」？**

- 如果是每次都要遵守的慣例（框架／套件、命名、禁改區），它該進 CLAUDE.md 或 rules，靠自動載入保證每次都在。
- 如果是特定任務才用得到的能力，封裝成 skill／subagent，靠觸發載入。
- 如果是「大部分時候用不到、但偶爾要查」的深度文件，就放 docs，靠明確引用載入——而你要負責在需要時把它引出來，或在 CLAUDE.md 裡留一句指路。

搞混這三者，會有兩種對稱的失敗：該自動載入的東西被丟進 docs，於是 agent 老是不照慣例；或反過來，把巨大的設計文件全塞進 CLAUDE.md，結果每次對話都被無關內容占滿 context，重要的鐵則反而被稀釋。

## 建立：讓該讀的被讀、該查的按需查

想清楚邊界之後，`.claude/` 的結構就有了明確的分工原則：**按「載入時機」來擺放，而不是按「這是什麼類型的檔案」。**

可以把稍後閱讀工具的 `.claude/` 重新整理成三個帶：

- **自動載入帶**：CLAUDE.md（專案慣例）＋ rules（行為準則）。內容要精、要短，因為它每次都占 context。
- **觸發載入帶**：skills 與 subagents。把「抓網頁內容並產摘要」這類重複的複雜任務收在這裡。
- **按需引用帶**：docs。深度設計筆記、schema、決策紀錄放這，平常不占 context。

關鍵一步，是在自動載入帶裡替按需帶留「路標」。既然 docs 不會自己進 context，那就在 CLAUDE.md 裡明白寫出：遇到什麼情況該去讀 docs 裡的哪份檔案。這樣自動載入的那一小段內容，就變成一張索引地圖——agent 每次開場都看得到，需要細節時知道去哪翻。這其實就是把「按需檢索」這件事，用最土法煉鋼的方式做出來：不是每次都把整座圖書館搬進來，而是留一份目錄，需要哪本再抽哪本（這正是「按需檢索」／RAG 最樸素的雛形，Day 26 會正式展開）。

順著這個原則，也正好把 Day 02 到 Day 05 的成果收斂起來。這五天各自做了一塊——CLAUDE.md、rules、skills／subagents、settings 與 hooks——把它們按載入時機歸位，其實就是把迴圈的幾個零件擺回定位：State/Memory（CLAUDE.md）負責「agent 會忘、repo 不會」，Skills 收起可重複的複雜任務，Guardrails（hooks／permissions）在關鍵動作前把關。零件各就各位，就自然長成一個結構清楚、可以整包 clone 的 starter——等於把迴圈的地基一次打包好。下次開新專案，clone 下來、把專案名稱和框架／套件改一改，地基就鋪好了，不用每次從零重來。

## 用 Claude Code 落地

下面是收斂後的 `.claude/` 目錄結構，直接可以當 starter 骨架：

```
.claude/
├── CLAUDE.md            # 自動載入：專案慣例 + 指向 docs 的路標
├── settings.json        # 自動載入：permissions 與 hooks（Day 05）
├── rules/               # 自動載入：行為準則（Day 03）
│   ├── naming.md
│   └── testing.md
├── skills/              # 觸發載入：可重複的複雜任務（Day 04）
│   └── fetch-and-summarize/
│       └── SKILL.md
├── agents/              # 觸發載入：subagents（Day 04）
│   └── summarizer.md
└── docs/                # 按需引用：深度文件，不自動進 context
    ├── summary-format.md
    ├── db-schema.md
    └── decisions/
```

重點在 CLAUDE.md 裡留的那段路標。它讓「按需引用帶」真的被引用得到：

```markdown
# 稍後閱讀 + AI 週報摘要工具（純 Go 後端服務）

## 框架／套件與慣例
- 純 Go 後端服務（不自帶前端）：Go 1.22+，HTTP 路由用 chi。
- DB 用 SQLite，driver 用純 Go 的 modernc.org/sqlite（不用需 cgo 的 mattn/go-sqlite3）。
- 查詢層用 sqlc 從 queries/*.sql 生成，生成的 internal/db/ 為禁改區；migration 用 goose。
- 命名與測試要求（Go 慣例、表格驅動測試）見 rules/。

## 深度文件索引（需要時再讀，不必每次載入）
- 動摘要格式或加新來源 parser 前，先讀 docs/summary-format.md。
- 碰資料表結構前，先讀 docs/db-schema.md。
- 想知道某個設計為何這樣定，查 docs/decisions/。
```

有了這段，前面那個早上的錯就不會再發生：agent 一開場就看到「加新來源 parser 前先讀 summary-format.md」，於是它會主動去把那份檔案讀進 context，格式自然對得上。差別不在檔案存不存在，而在有沒有一條路把它引進來。

載入策略講白了就三句話：鐵則放能自動載入的地方、能力封裝成觸發載入、深度文件放 docs 並在 CLAUDE.md 留索引。

## 對照概念：記憶管理與按需檢索

把場景拉高一層看，這件事對應到 agent 設計裡兩個常被一起討論的概念。

一個是**記憶管理**。agent 的 context window 是有限且昂貴的資源，不可能把所有知識都常駐其中。所以要區分「長期記憶」——每次都該在場的核心事實（對這個專案來說就是 CLAUDE.md 那一小段慣例與索引）——和「隨用隨取」的外部知識。把什麼放進常駐記憶、什麼留在外面，本身就是一種取捨。

另一個是**按需檢索**。不是把整個知識庫塞進 context，而是保留一個入口，在需要時才去把相關片段取進來。用 CLAUDE.md 的路標加上 docs 目錄做的，正是這個概念最樸素的版本：目錄當索引，內容按需抽取。等到專案大到人工路標不敷使用，這個思路會自然長成更完整的檢索機制，但底層的判斷是同一個——**context 是稀缺的，什麼該常駐、什麼該按需取，得刻意決定。**

## 今日實踐任務

今天替稍後閱讀工具收尾地基，把 Day 02–05 收斂成一個可 clone 的 starter。

1. 按「載入時機」重排 `.claude/`：分出自動載入帶（CLAUDE.md、settings.json、rules）、觸發載入帶（skills、agents）、按需引用帶（docs）。
2. 檢查每份 docs——它現在有沒有辦法被引用到？在 CLAUDE.md 補一段「深度文件索引」，替每份重要的 docs 留一句路標。
3. 把整個 `.claude/` 抽成 starter 範本：移除專案專屬內容，換成佔位說明，讓別的 Go 後端專案 clone 下來改幾個名字就能用。

驗收標準：

- 開一個新對話，只問「這專案的摘要格式慣例是什麼」，agent 能循著 CLAUDE.md 的索引主動讀到 docs 裡的答案，而不是憑空瞎猜。
- CLAUDE.md 本身夠短——掃一眼就知道專案長什麼樣、細節去哪查，而不是把設計文件整段搬進去。
- starter 目錄能在另一個空專案裡 clone、改名、直接沿用，地基一次到位。
