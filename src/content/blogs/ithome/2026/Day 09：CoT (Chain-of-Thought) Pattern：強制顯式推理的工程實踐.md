昨天的 ReAct 給了 Agent 一個對外的迴路：每一步推理都踩在真實的 Observation 上，用環境的回饋校正下一步。ReAct 之所以不太會亂講，是因為外面有個現實會打它的臉——`grep` 找不到、測試跑不過，思路就會被拉回來。

但這裡藏著一個前提：任務必須「碰得到環境」。如果沒有外面呢？當問題是「給定這段狀態轉移，這個 DP 解法對不對？」「這個複雜度推導有沒有漏掉主導項？」這類純邏輯推導時，根本沒有外部世界可觀察，ReAct 會空轉——它少了 Observation 那條腿。這時候我們需要的不是對外的迴路，而是對內的紀律：強迫模型把腦中那條本來「一閃而過」的推理鏈攤開、寫成可檢視的顯式步驟。這就是 CoT (Chain-of-Thought)。

## 1. 為什麼需要 CoT？

先看一道小學程度的題目：「原價 1200，先折抵 100，再打八折，最後多少錢？」

模型很容易脫口答 860。它記住了「打折題的答案形狀」，於是直接 1200 × 0.8 - ... 一步到位，把運算順序搞混了。正解是 (1200 - 100) × 0.8 = 880。

這種錯誤的可怕之處在於：它錯得很有說服力。我們可以稱它為「自信的草率」——模型沒有在哪一步明顯卡住，而是把該分開的兩步壓成一步，跳過了「先折抵、再打折」這個必須顯式走過的順序。

要理解為什麼「攤開來寫」能治這個病，得先抓住一個核心事實：

> LLM 沒有一塊獨立的「思考區」可以打草稿。它能用來推理的唯一空間，就是它正在生成的那串 token 本身。沒寫出來的推理步驟，等於沒發生。

換句話說，顯式推理 token 就是模型的「草稿紙」。當你強迫它先寫出「第一步：折抵後是 1100；第二步：1100 打八折是 880」，這些中間 token 會回頭成為後續生成的上下文，把最終結論錨定在已經寫出的推導之上。沒有這張草稿紙，模型只能憑直覺一步到位，跳步出錯的機率大增。

CoT 的工程價值，就是把這張草稿紙從「模型高興才打」變成「協議強制它打」。

## 2. CoT 的核心機制：線性、純內在、不碰環境

CoT 的形態比 ReAct 單純得多，它有三個特徵：

- **線性**：一條從前提到結論的推理鏈，每一步只引用前面已寫出的步驟，不分叉、不回溯。
- **純內在**：所有推導都在模型生成的 token 裡完成，不呼叫工具、不讀檔、不跑測試。
- **不碰環境**：沒有 Observation，因此答案的正確性完全由「題目前提 + 推導本身」決定。

它的骨架可以畫成這樣：

```text
問題
  ↓
步驟 1   （只依據題目前提）
  ↓      （引用步驟 1）
步驟 2
  ↓      （引用步驟 1、2）
步驟 3
  ↓
結論     （建立在前面所有步驟之上，全程不查外部）
```

對照 Day 08 的 ReAct 三段式循環會發現：ReAct 的每個 Thought 緊扣上一個 Observation（外部事實），CoT 的每個步驟則緊扣上一個步驟（內部推導）。一個向外要證據，一個向內要紀律。

## 3. .claude/ 中的協議實作

跟 Day 08 一樣，我們把這條「強制顯式推理」的紀律「編譯」進規則檔，讓 Agent 在做純邏輯推導時不得跳步、不得在推導完成前就丟 code。

### .claude/rules/cot-protocol.md

```markdown
# CoT 強制顯式推理協議

適用場景：純邏輯/演算法推導，不需要與外部環境（檔案、API、測試）互動的任務。

## 1. 強制輸出推導鏈

在給出任何結論或 code 之前，MUST 依序輸出以下五個區塊，缺一不可、不得跳步：

### [1] 問題拆解
- 用自己的話複述題目要解什麼，標出輸入、輸出、限制。
- 把大問題切成可獨立處理的子問題。

### [2] 狀態與不變量
- 定義解法過程中要追蹤的「狀態」是什麼。
- 寫出貫穿全程的不變量 (invariant)——那個「每一步都必須成立」的性質。
- 若採用某個技巧（DP、雙指針、貪婪…），MUST 寫出它成立的前提條件。

### [3] 邊界條件
- 列出至少 4 種極端輸入並逐一說明預期行為：
  空輸入、單一元素、最大規模、特殊值（負數 / 0 / 重複 / 溢位）。

### [4] 複雜度分析
- 推導時間複雜度與空間複雜度，寫出推導理由（不是只給結果）。
- 對照題目限制，確認此複雜度可接受；若不可接受，回到 [1] 重想。

### [5] 結論 / code
- 只有在 [1]–[4] 全部完成後，才允許輸出最終結論或程式碼。
- code MUST 與前四步的推導一致；若 code 與推導矛盾，以推導為準並修正。

## 2. 停止與品質要求

- 任一區塊若無法填寫（例如想不出不變量），MUST 停下並明說「此處推不下去，原因是…」，
  NEVER 用模糊話術硬湊過去。
- NEVER 在 [4] 之前出現任何最終 code。發現自己想跳步直接寫 code，視為違規，回到對應區塊。
- 每一步的結論只能建立在「題目」與「前面已寫出的步驟」上，NEVER 引入未經推導的假設。
```

這份協議的本質，是把「先想清楚再動筆」這件人類資深工程師的習慣，變成模型不得繞過的輸出格式。

### 一個誠實的提醒：rules 屬於 context 層

從 Day 01 與 Day 03 一路講到 Day 08，有一條原則必須在這裡再強調一次：rules 是寫進 context 的「叮嚀」，不是「閘門」。

模型每次生成時都會讀到 cot-protocol.md，於是「傾向」按五步走——但 rules 並不能在引擎層面保證它一定照做。模型可能在某次回應裡偷懶跳過 [3] 邊界條件，直接跳到 code，而 rules 本身攔不住。這跟 Day 08 對 react-protocol 的提醒是同一回事：context 改變的是模型的意願，不是它的能力邊界。

> rules 負責「讓模型想這麼做」，hooks 負責「讓模型只能這麼做」，兩者分工、缺一不可。

如果你要的是硬性把關，就得搭配 hooks。例如在 Agent 即將輸出 code 前攔截，掃描這次回應裡有沒有出現「複雜度分析」這個區塊標題；找不到就擋下，要求補完再放行。rules 讓它「想這麼做」，hook 讓它「只能這麼做」。這篇先把 rules 寫扎實，hook 留到今日實踐任務的進階題去掛。

## 4. 一段真實的 CoT Trace

光看協議很抽象，我們拿一道貼合演算法陪練 side project 的題目實際跑一遍：

> 給一個整數陣列（**可能含負數**）和一個目標值 k，求「和恰好等於 k」的最長連續子陣列長度。

注意這個「含負數」的伏筆——它正是省略推理最容易踩的坑。以下是 Agent 按 cot-protocol 輸出的軌跡：

```text
[1] 問題拆解
    輸入：整數陣列 nums（含負數）、目標和 k。
    輸出：和恰好為 k 的最長連續子陣列的長度（找不到回 0）。
    子問題：如何在掃描中快速判斷「是否存在某個起點，使該段和為 k」。

[2] 狀態與不變量
    狀態：前綴和 prefix[i] = nums[0..i] 的總和。
    關鍵推導：子陣列 (j+1..i) 的和 = prefix[i] - prefix[j]。
              要它等於 k，等價於 prefix[j] = prefix[i] - k。
    前提檢查：能不能用滑動視窗？滑動視窗要求「擴張視窗時和單調遞增」。
              但陣列含負數 → 加入元素可能讓和變小 → 單調性被破壞 →
              滑動視窗不成立。改用「前綴和 + 雜湊表」。
    不變量：雜湊表 seen 記錄「每個前綴和第一次出現的索引」，
            因此查到的 prefix[i]-k 一定是最早的起點，才能取到最長長度。

[3] 邊界條件
    - 空陣列：回 0。
    - 單一元素且等於 k：回 1。
    - 全程無解：回 0。
    - 整段恰好為 k：要能取到（所以 seen 需預置 {0: -1}，代表空前綴）。

[4] 複雜度分析
    單趟掃描，每步做 O(1) 的雜湊查詢與寫入 → 時間 O(n)。
    雜湊表最多存 n+1 個前綴和 → 空間 O(n)。
    對照常見限制 n ≤ 1e5，O(n) 可接受。

[5] 結論 / code
```

```python
def longest_subarray_sum_k(nums, k):
    seen = {0: -1}          # [2] 不變量：前綴和首次出現的索引；預置空前綴
    prefix = 0
    best = 0
    for i, x in enumerate(nums):
        prefix += x         # [2] 維護前綴和 prefix[i]
        need = prefix - k   # [2] 找 prefix[j] == prefix - k 的最早 j
        if need in seen:
            best = max(best, i - seen[need])  # [3] 取最長長度
        if prefix not in seen:
            seen[prefix] = i  # [2] 只記首次，保證起點最早
    return best             # [3] 找不到自然回 0
```

整段 trace 的價值，全在 [2] 那一句「含負數 → 單調性被破壞 → 改用前綴和」。如果模型跳過顯式推理、憑直覺看到「子陣列和」就套滑動視窗，會在含負數的測資上默默出錯。把前提檢查攤到檯面上，這個坑就被草稿紙接住了。

## 5. CoT vs ReAct vs ToT：三種推理模式的定位

Day 08 已經並排比較過一次，今天從 CoT 的視角把三者的分工講透：

| 模式 | 結構 | 是否碰環境 | 對治的問題 | 適用場景 |
| --- | --- | --- | --- | --- |
| CoT (Chain-of-Thought) | 線性單鏈 | 否 | 跳步、自信的草率 | 純邏輯/數學/演算法推導 |
| ReAct (Reasoning + Acting) | Thought→Action→Observation 循環 | 是 | 幻覺、脫離真實狀態 | 需讀檔、查 API、跑測試的工程任務 |
| ToT (Tree-of-Thoughts) | 樹狀多分支 + 評估剪枝 | 視實作而定 | 過早承諾單一路徑 | 搜索空間大、需試錯比較的問題 |

這三者不是互斥的選項，而是不同粒度的組裝。最值得記住的一點：**CoT 可以內嵌進 ReAct 的 Thought**。當 ReAct 走到某個需要深思的 Thought 時，可以在那個 Thought 內部跑一小段 CoT，想清楚了再決定 Action：

```text
Thought:
  我要判斷該不該重跑測試。先在腦中推一遍：
  [步驟1] 上次失敗是 import 錯誤，不是邏輯錯。
  [步驟2] 我剛改的是 import 路徑，理論上已修正。
  [步驟3] 但我還沒確認 tests/ 目錄的實際結構。
  結論：不該瞎重跑，應先確認測試檔位置。
Action: grep -rn "from src" tests/
Observation: ...
```

所以可以這樣收束：CoT 是最小的推理單元——「把一條思路寫清楚」對抗跳步；ReAct 是「讓思路踩在真實證據上」對抗幻覺；ToT 是「同時想好幾條思路再挑」對抗過早承諾。ReAct 與 ToT，本質上是 CoT 這個最小單元的兩種不同組裝方式。

判準也很乾脆：問題的答案能不能「只靠題目給的前提」推出來？能，用 CoT；需要查資料或試用才知道，用 ReAct；需要比較多種可能再挑，用 ToT。

## 6. Critical Thinking：CoT 的限制

CoT 強大，但有它治不了的病，務必看清楚：

- **它治「跳步」，不治「前提錯」。** CoT 只保證推理鏈內部自洽，不保證起點正確。如果第一步的前提就錯了，整條鏈會忠實地、條理分明地順著錯下去——而且寫得越詳細越有說服力，越容易騙過審閱的你。這種「自洽地錯」是 CoT 最危險的盲點。要對治前提錯，得靠 Self-Reflection（Day 13 會談）回頭質疑前提，或靠 ReAct 的外部驗證把前提丟去現實裡撞一下。
- **token 成本。** 強制顯式推理意味著每個結論前面都掛著一大段推導 token，輸出變長、延遲變高、費用上升。對「1 + 1 等於幾」這種一眼可答的問題硬套五步協議，是殺雞用牛刀。CoT 該用在推導真的有深度、跳步真的會出錯的地方。
- **不碰環境，所以資訊只能來自題目。** CoT 沒有 Observation 這條腿，它無法得知題目沒寫的事實。一旦任務需要「現在這個檔案長怎樣」「這個 API 回傳什麼」，CoT 就無能為力——那是 ReAct 的領域。誤把需要環境互動的任務塞給 CoT，模型就會開始憑記憶捏造，回到我們 Day 08 想避開的幻覺老路。

一句話：CoT 讓推理「過程」可檢視，但不替推理的「前提」與「素材」背書。
