在過去兩年，AI 應用的開發往往聚焦於「Prompt Engineering」。我們花費大量時間調整 System Prompt，試圖讓模型在單次呼叫中輸出完美的結果。然而，當我們試圖解決複雜、長週期的軟體工程任務（如重構一個模組、自動化 CI/CD、複雜數據處理）時，這種「單次 Prompt」的方法很快就會達到極限——我們稱之為 **「複雜度崩潰 (Complexity Collapse)」**。

## 1. 為什麼單點推理 (Single-shot) 會失敗？
單純依賴 LLM 的湧現能力就像是在賭博，缺乏工程上的可預測性。在處理長任務時，我們常面臨三大挑戰：
*   **上下文衰減 (Context Decay)：** 模型在執行到第 10 個步驟時，往往已經模糊了最初的設計約束。
*   **不可觀測性 (Lack of Observability)：** 如果任務失敗，開發者通常只看到最終錯誤，而無法洞察 Agent 在哪個決策節點發生了偏離。
*   **邏輯漂移 (Logic Drift)：** 沒有嚴格的流程紀律，AI 會在多任務間產生思維發散，導致代碼庫結構逐漸變得混亂。

## 2. 什麼是 Workflow Pattern？
Workflow Pattern 是一種**「將認知處理過程結構化」**的架構語言。它不是單純的 Prompt，而是一套定義明確的**邏輯拓撲 (Logical Topology)**。一個成熟的 Pattern 定義了三個核心元素：
1.  **節點 (Nodes)：** 執行的最小單元（可能是 AI 推理，也可能是具體的 Skill）。
2.  **邊 (Edges)：** 資料流與控制流，規定了前一步驟的產出如何成為下一步驟的輸入。
3.  **邊界 (Boundaries)：** 定義了任務成功、失敗、以及何時應進入 Human-in-the-loop 的判定條件。

## 3. 本系列的核心執行協議：Claude Code
在接下來的 30 天中，我們將統一使用 **Claude Code** 作為核心執行引擎。我們不將其視為一個 Chatbot，而是一個 **CLI-based Agentic Runtime**。它的價值在於：
*   **環境共生：** 它直接運作在你的檔案系統上，讀取 `git` 歷史、執行 `bash` 腳本、分析 `AST` 樹。
*   **協議驅動：** 我們透過 `.claude/` 目錄下的規則檔（Rules）來約束其行為，讓它從一個「聊天機器人」轉型為一個「嚴謹的開發者助理」。

## 4. 30 天實作路線圖
為了讓開發者能從零建構一套 Agent 協作協議，我們規劃了以下階段：
*   **第一階段：環境與目錄協議 (Day 01-07)** - 深入剖析 `.claude/` 目錄結構，從基礎配置到架構化執行。
*   **第二階段：決策與推理模式 (Day 08-14)** - 建立 Agent 的思考骨架（ReAct, CoT, ToT 等）。
*   **第三階段：協作與對抗模式 (Day 15-20)** - 多 Agent 的層級管理與博弈（Supervisor, Debate 等）。
*   **第四階段：韌性與執行模式 (Day 21-26)** - 處理長任務、容錯與狀態持久化（Fallback, State-Management 等）。
*   **第五階段：生態系統與進化 (Day 27-30)** - 整合測試驅動開發、文檔同步與私有模式語言。

## 5. 今日思考 (Critical Thinking)
在進入目錄結構的研究前，請先建立一個關鍵觀念：**Agent 的智商不是由模型決定的，而是由你的 Workflow 架構決定的。**

一個設計良好的 Workflow Pattern，能夠將一個中等能力的模型，提升到解決複雜軟體工程任務的水準。準備好開始建構你的 Agentic 執行環境了嗎？
