## 從一段對不起來的去重邏輯說起

昨天把地基收完，記憶、規則、技能、守門，加上 context 邊界的釐清，該讀什麼、該按需查什麼都各就各位了。環境鋪好之後，接下來這幾天要處理的是另一件事：怎麼讓 agent 在動手之前先把事情想清楚。今天從最單純的一種問題開始，一種完全在腦子裡就能算完、不需要碰任何檔案或工具的純邏輯題。

情境是這樣。稍後閱讀工具的週報要把一週內存下的文章做去重，然後照「重要度」排序。去重的規則我想得挺細：同一個 URL 只留一篇；但同一篇文章可能有不同來源的連結（原站、轉載、AMP 版本），這種要視為同一篇；已經讀過的文章排序時往後放，但如果它被重新標記為「想再讀」，就拉回來。排序則是先按「未讀優先」，同組內再按存入時間新到舊。

我把這串規則講給 agent 聽，請它直接寫個 `dedupeAndRank` 函式出來。它很爽快地給了一段程式碼，看起來也很像那麼回事。問題是拿實際資料一跑，AMP 版本和原站被當成兩篇留了下來，而且「想再讀」的那條規則整個被忽略——已讀就是往後排，不管有沒有被重新標記。

我盯著那段 code 看了一會兒才明白問題出在哪。它根本沒有把我那幾條規則之間的關係想過一遍就開始寫了。URL 正規化（把 AMP、轉載對應回原站）是去重的前提，這步沒做，後面全錯；而「已讀」和「想再讀」是有優先順序的兩個狀態，它直接把兩個 if 平列，後面那條永遠壓不過前面那條。這不是它不會寫程式，是它跳過了「先把規則之間的依賴關係理清楚」這一步，憑著對題目的第一印象就下筆了。

## 判斷：什麼時候該逼它先想清楚

這類問題有一個很清楚的共同特徵：答案完全由題目本身決定，不需要去外面查任何東西。去重排序的邏輯、一段演算法的邊界條件、一個狀態機該有哪些轉移、某個計算的複雜度，這些都是封閉式的。你把規則講清楚，正確答案就已經被規則鎖死了，剩下的只是把它一步步推導出來。

麻煩在於，模型面對這種題目時很容易「跳步」。它讀完題目，腦中浮現一個大概的方向，就直接跳到最後的答案，中間那些本該逐條檢查的推導被省略了。多數時候第一印象是對的，所以你會覺得它很行；但只要題目稍微繞一點，有幾條規則彼此有依賴、有先後、有例外，跳步就會在某個中間環節悄悄出錯，而最後產出的 code 表面上完全正常，你不細看根本抓不到。

判斷準則其實很簡單：**當一個問題需要好幾個中間步驟才能推到答案，而且每一步都建立在上一步的結論上，就值得逼它把推理攤開。** 特別是這幾個訊號同時出現時——規則之間有依賴或先後、有需要特別處理的邊界與例外、答案錯了但你一眼看不出錯在哪——那就別讓它一步到位，先要它把推導過程寫出來。

反過來，什麼時候不必這樣做？兩種情況。一種是問題太簡單，一步就到，硬要它列推導反而囉嗦。另一種更重要，也是這三天要分清楚的重點：**只要這個任務需要去碰外面的世界，它就不是今天這個場。** 去讀一個檔案、跑一個指令、查一個 API 回傳什麼——這些答案不在題目裡，在環境裡。純推理再嚴謹也生不出環境的事實，你逼它在腦中「想」一個檔案的內容，它只會很有條理地瞎猜。那是明天 ReAct 要處理的問題。今天談的手法只對「答案已經藏在題目裡、只是需要一步步挖出來」的封閉式問題有效。

## 建立：把推導和實作拆成兩步

要治跳步，核心動作就一個：**在它產 code 之前，先強制它把推理過程顯式寫出來。** 但光說「請你想清楚」沒用，你得指定它要想哪些東西，否則它還是會給你一句「我仔細想過了」然後照樣跳步。

我的做法是把「推導」和「實作」拆成兩個獨立的步驟，中間卡一道關。第一步只准它做推理，明確要求它交出這幾樣東西：

- **列出所有狀態與輸入**：這題涉及哪些狀態、哪些欄位、哪些可能的組合。以去重排序來說，就是「已讀／未讀」「是否被標記想再讀」「URL 的各種變體」這些。
- **理清規則之間的依賴與先後**：哪一步是哪一步的前提，哪些規則之間有優先順序。像 URL 正規化必須在去重之前，「想再讀」必須能覆蓋「已讀」的排序。
- **標出邊界與例外**：空清單、只有一篇、全部同組、同時已讀又想再讀，這些邊角情況各該怎麼處理。
- **推估複雜度**：資料量大時這個做法會不會爆，有沒有更省的走法。

這一步結束時，它產出的是一份「解題說明」，不是程式碼。我先讀這份說明，確認它對規則之間的關係理解正確——如果它在這裡就把「想再讀覆蓋已讀」漏掉了，我當場就能揪出來，成本是零。確認無誤，才進第二步：拿著這份已經對過的推導去寫實作。

這種「先推導、確認、再實作」的兩段式，本質上就是把一個大請求切成一條有順序的小步驟鏈。每一步的產出是下一步的輸入，而且在關鍵的接縫處（推導完成、要動筆之前）留一個檢查點。好處很直接：錯誤在最便宜的階段就被攔下，而不是等它寫完一整段 code、你 debug 半天才發現病根在最初的理解。

## 用 Claude Code 落地

把上面的流程固定下來，最省事的是寫成一條 rule，讓 agent 碰到這類題目時自動走兩段式，不用我每次都重講。

在 `.claude/rules/` 底下放一份 `reasoning-first.md`：

```markdown
# 純邏輯／演算法題：先推導再實作

適用：實作一段有多條規則、有邊界條件、有狀態轉移，
或需要選演算法的邏輯時（例如去重、排序、狀態機、解析器）。
不適用：需要讀檔、跑指令、查 API 實際回傳的任務——那走 ReAct。

## 硬性流程：分兩步，不准跳步

### 第一步：只做推導，不要寫任何實作程式碼

先輸出一份「解題說明」，至少包含：

1. 列出所有輸入、狀態與其可能組合。
2. 逐條列出規則，並標明規則之間的依賴與優先順序
   （哪步是哪步的前提、哪條會覆蓋哪條）。
3. 標出邊界與例外情況，各自的預期行為。
4. 用一兩組具體輸入手動走一遍，寫出預期輸出。
5. 推估時間／空間複雜度，說明有無更省的做法。

寫完停下，等我確認這份推導沒問題，再進第二步。

### 第二步：依已確認的推導寫實作

- 實作必須對得上第一步的每一條規則與邊界。
- 若實作時發現推導有漏，回第一步補，不要私自改邏輯。
```

重點在第一步結尾那句「寫完停下」。它把推導和實作之間切出一道人可以介入的縫，讓我在最便宜的時間點看一眼它到底有沒有理解對。如果你想更自動一點，也可以不停下，讓它自己接著寫，但要求它在第二步開頭先逐條複述第一步的規則、再對照著實作——重點是那份顯式推導一定要先存在，跳步才無所遁形。

真的要臨時用、不想寫 rule 的話，一句 prompt 也能達到同樣效果：

```
先別寫 code。第一步，把這題的所有狀態、規則之間的先後與
覆蓋關係、邊界例外都列出來，再用一組實際輸入手動走一遍給我看。
我確認過推導，你再寫實作。
```

拿這套重跑一次去重排序，第一步它就自己把「URL 正規化要先於去重」「想再讀要能覆蓋已讀排序」寫進了依賴關係裡，還主動問我「同時已讀又想再讀時，時間排序要用原始存入時間還是重新標記的時間」——這正是我原本沒講清楚、它上次直接猜錯的那個例外。推導攤開之後，實作幾乎一次就對了。

## 對照概念：思維鏈與步驟拆解

把手法拉高一層看，這對應到一個談推理時常提到的概念：讓模型在給出答案前，先把中間的推理步驟一步步生成出來，而不是直接跳到結論。它的效果來自一件挺樸素的事——模型是順著已經生成的內容往下接的，當它先寫出了正確的中間步驟，接下來的每一步就有了可以踩的立足點，而不是懸空去猜最後答案。逼它把「想」的過程外顯成文字，等於強迫它真的走完每一步，而不是靠第一印象抄捷徑。

要特別界定清楚的是它的邊界：這種推理完全發生在模型內部，不牽涉任何外部互動。它不讀檔、不跑工具、不觀察環境回饋，就是把題目給定的資訊在腦中一路推導到底。所以它只對「答案已經被題目鎖死、缺的只是推導」的封閉式問題有效——邏輯題、演算法、數學、規則推演。一旦問題的答案需要去環境裡才問得到，純內在推理就使不上力了。

而把「推導」和「實作」拆成兩個前後相接的步驟、在中間設檢查點，則是另一個相鄰但不同的概念：把一個複雜請求拆成一條有序的子步驟，讓每一步聚焦一件事、上一步的產出餵給下一步。這兩個概念今天是搭在一起用的——用步驟拆解搭出「先推導、後實作」的骨架，用思維鏈填滿「推導」那一步的內容。

## 今日實踐任務

今天替稍後閱讀工具真正把去重與排序邏輯做對，順便量一下「有沒有先推導」到底差多少。

1. 在 `.claude/rules/` 放一份 reasoning-first.md，寫清楚兩段式流程與適用／不適用範圍。
2. 先用「省略推理」的方式跑一次：直接請 agent 寫 `dedupeAndRank`，把規則一次講完、要它直接給 code。記下它的產出。
3. 再套 rule 用兩段式跑一次：先要推導、你確認、再實作。
4. 兩份產出都拿同一組測資去跑，包含這幾個刁鑽案例：AMP／轉載連結對應同一原站、同時「已讀」又「想再讀」、空清單、整組同來源。

驗收標準：

- 兩段式版本在第一步的推導裡，就明確寫出了規則的依賴與覆蓋關係（URL 正規化先於去重、想再讀覆蓋已讀排序），而不是等寫完 code 才發現漏。
- 兩份產出跑同一組測資，記下各自答錯幾個案例。省略推理的版本錯誤數應明顯較高——把這個差距記下來，它就是「先想清楚再下 code」對這類題目的實際價值。
- rule 的適用範圍寫得夠明確：讀者一看就知道這招只治封閉式邏輯題，碰到要讀檔／跑指令的任務不該套它。

## 明天 Day 08

今天這套推理只在腦子裡跑，題目給定的資訊足夠它一路推到底，所以逼它攤開推導就能治跳步。但它有個硬邊界：一旦答案不在題目裡、而在真實世界裡——要去讀一個檔案、跑一個指令、看某個 API 到底回傳什麼——純內在推理就開始瞎猜，而且會猜得很有條理，讓你更難發現它其實是在編。明天用 ReAct 補上這一塊：讓 agent 邊查邊做，每一步都先觀察環境、再根據觀察決定下一步，把推理踩在證據上而不是印象上。
