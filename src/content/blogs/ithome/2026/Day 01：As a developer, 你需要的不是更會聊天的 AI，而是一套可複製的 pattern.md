## 一個每天都在發生的場景

昨天你跟 AI 一起改了一個模組，過程很順。你先跟它交代了專案用什麼框架、測試放哪、哪幾個檔案不准動，它照做了，結果不錯。

今天你開了一個新對話，想接著改隔壁那個模組。它掃過專案、大方向沒問題——框架、目錄它讀得出來。但那些你昨天只是「講」給它聽、沒寫進任何檔案的約定：哪幾個檔案不准動、某個地方為什麼要這樣寫、之後打算怎麼換，它一概不知道，於是又按最常見的預設猜了一輪。你嘆了口氣，把昨天那串交代再打一次。它照樣可以做得不錯，但你做的事情，其實是「重新哄它一次」。

一開始你可能以為問題出在「prompt 下得不夠好」。於是去看了很多下 prompt 的技巧，收藏了一堆「超強咒語」。它們確實有用，但只有那一次有用。下次換個任務、換個對話，又回到原點，重新拼湊那串話。真正缺的，其實不是一句多完整的 prompt，而是**一套不用每次重講、結果可以複製的運作方式**。

## 問題不是「不夠聰明」，是「不可複製」

現在的模型已經很強了。它會 thinking、會查資料、會呼叫工具、會分步驟推理。單看每一次的表現，它常常超出你的預期。

但「單次表現好」跟「能穩定依賴它」是兩回事。開發是一件重複的事：同一種問題會反覆出現，同一套慣例要一直遵守，同一類任務下週還會再來一次。如果每一次都得靠你當下的心情、當下記得多少、當下 prompt 下得多完整，那你得到的就是一個運氣成分很高的協作者，而不是一個工具。

你要的其實很簡單：同一種情境，下一次能得到差不多水準的結果，而且不必從零交代。這在軟體工程裡有個老名字，叫**可複製性**。我們對建置流程、對測試、對部署，早就要求可複製了，卻在跟 AI 協作這件事上，退回到「每次手動、每次憑感覺」。

## 換一個問法：你今天在建哪一個迴圈

所以這個系列想把問題換一個問法。

不是「怎麼把 prompt 寫得更漂亮」，而是：**你今天在建、在收緊哪一個迴圈？** 這個問法背後，是社群裡正在發生的一個轉變。Anthropic Claude Code 負責人 Boris Cherny 說過：

> My job is to write loops.

開發者 Peter Steinberger 也說：

> You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents.

這兩句話指向同一件事：你能操控的槓桿，正從「一次把話講好」往上移到「設計一個能反覆跑、會自己收斂的過程」。

因為一個 agent 就是一個迴圈：**act → observe → verify → repeat**——動手做、看結果、驗證對不對，不行就再來一輪，直到目標達成。它跟你合作的效果好不好，其實不取決於你這次 prompt 背得多熟，而取決於你替它工程化了什麼樣的回饋迴路：回饋多快回來、多真實、能不能讓它自己收斂、什麼時候該停、由誰來驗。這些才是你真正能設計的東西。

於是這個系列的每一天，都會帶著一個具體的、開發時真的會遇到的問題進來——AI 沒讀檔就開始瞎猜、大重構散落在十幾個檔案、上線後湧進一堆雜亂的 feedback——然後一起回答：**這個場景，你該建哪一個迴圈，用 Claude Code 怎麼把它落地成可複製的設定？** 它可能是一份寫好的 rule、一個封裝好的 skill、一道自動的 hook、或是幾個 sub-agent 組成的產線；但無論長成什麼樣，它們做的事情是同一件：把某一段「想→動→看→修」固定下來，讓你下次不必重想、直接拿來用。

至於那些通用的 pattern 名稱（像 ReAct、Supervisor、Reflection 這些），會放在每篇的括號裡當對照詞彙，方便你之後查到正式術語。但先說一句：它們其實都是迴圈技術——ReAct 是最小的「做一步、看一步」迴圈，Reflection 是「生產者—檢查者」的迴圈，TDD 是紅綠迴圈。名稱不是主角，你當下在建的那個迴圈才是。

## 一個會真的長大的範例專案

要讓「情境 → 迴圈」不流於舉例，最好的辦法是全程用同一個專案。

所以整個系列會綁在同一個範例專案上：一個「稍後閱讀加上 AI 週報摘要」的**純 Go 後端服務**。它的功能講起來很單純——使用者透過 API 把想看的文章連結存進來，服務幫忙抓內容、呼叫 LLM 產出單篇摘要，到了週末再把這一週存的東西彙整成一份週報，用 API 或 Email 輸出。它不自帶前端，重心整個放在後端這條資料流上。

語言的話，就直接用 Go，HTTP 路由用 chi，資料庫用 SQLite（driver 用純 Go 的 modernc.org/sqlite，不必碰 cgo），查詢層用 sqlc 從 SQL 檔生成型別安全的存取層，migration 用 goose，測試用 go test 走表格驅動。目錄大致是 `cmd/server/` 放進入點，`internal/` 底下按 api、service、fetcher、summary、report、db 分層。

選這個題目，單純只是因為它會把三種階段的問題全踩過一遍：

- 一開始它是個一人 MVP。你要獨自把核心功能做出來（存連結、抓內容、單篇摘要），這時候最難的是「怎麼讓 AI 照你的規矩、想清楚再動手」，也就是建起一個最小、踩在真實回饋上的內迴圈。
- 接著功能變多，一個人顧不來每個環節，開始需要把工作拆給不同分身、加上審查，讓交付變成一條可靠的產線，也就是把單一迴圈擴成多角色、能平行跑的迴圈。
- 最後它上線了、有了真的使用者。這時候要面對的是會失敗的外部呼叫、爆掉的 context、湧進來的回饋，以及「下一個功能該做什麼」——這一段要讓迴圈在維運中站得住、最後自己跑起來。

換句話說，這個服務在 30 天裡遇到的每一件麻煩事，剛好對應到後面每一天要收緊的那個迴圈。如果你手上剛好有想做的真實後端專案，把名字換掉就好，底下這三個階段的結構是一樣的。

## 這 30 天的地圖

整個系列分成三個 Part，對應「把一個範例專案一路包進一個能自己跑的迴圈」的三個層次。

第一個是**地基與內迴圈**（Part 1）。這一段在鋪迴圈的零件：用 Claude Code 的 memory、rules、skills、hooks／permissions 這幾個機制，把「讓 agent 記得慣例、照著規矩、危險動作被擋下來」固定下來；再加上幾種推理模式，讓它在動手前先想清楚。這一段結束時，你手上會有一個最小但真實的內迴圈——「想 → 動 → 看 → 修」踩在專案的實際回饋上，而不是每次重講的口頭交代。

第二個是**多角色與平行迴圈**（Part 2）。當單一 agent 顧不來、任務開始散開時，我們用 sub-agent 把幾個角色組合起來——有人拆派、有人執行、有人專門挑錯、有人負責選型——並讓它們能平行跑、彼此隔離。這一段是把一個人的內迴圈擴成一條多角色的生產線。

第三個是**維運與自主迴圈**（Part 3）。專案上線之後，重點從「把功能做出來」轉到「讓它穩定活著並持續長大」：處理會失敗的呼叫、把爆掉的 context 收好、把湧進來的使用者回饋彙整分類排優先、發想下一批功能，最後用 automations 把「發現工作 → 實作 → 驗證 → 更新狀態」串成一個能自己跑的自主迴圈。驗證不是獨立的一段，而是一路內建在迴圈裡——從自我檢查、生產者對審查者、到紅綠測試，最後由獨立的 verifier 與回歸檢查把迴圈閉起來，出錯就擋下。

這三段之間是遞進的。內迴圈沒建好就談多角色，會發現分身之間各說各話；沒有可靠的生產線就談上線維運，會被回饋淹沒。所以我們一天一天來，跟著那個 Go 服務的成長順序走。

## 接下來 30 天，你可以怎麼跟著做

這個系列可以只當讀物，但如果你願意動手，收穫會差很多。

建議是：開一個你自己的範例專案，可以就是那個稍後閱讀服務，也可以換成任何你真正想做的後端專案。每天讀完一篇，就把當天那個迴圈套進你的專案，落地成一份可複製的設定——一份 rule、一個 skill、或一組 sub-agent。三十天下來，你收緊過的每一個小迴圈會慢慢疊起來，最後合成一個能替你自己跑的開發迴圈，下次直接拿來用。

明天 Day 02，我們從地基的第一塊零件開始，處理最惱人的那件事：AI 每開一次新對話就忘記專案慣例、要你重講一遍。我們會把該記住的東西寫進 CLAUDE.md 與 auto memory，補上迴圈裡的 State/Memory 這一塊，讓 agent 一啟動就對得上你的專案，不必再從零哄它一次。
