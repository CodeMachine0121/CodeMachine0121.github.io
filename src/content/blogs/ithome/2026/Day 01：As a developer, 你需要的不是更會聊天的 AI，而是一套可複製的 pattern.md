## 先講一件我自己常發生的事

昨天我跟 AI 一起改了一個模組，過程很順。我先跟它交代了專案用什麼框架、測試放哪、哪幾個檔案不准動，它照做了，結果不錯。

今天我開了一個新對話，想接著改隔壁那個模組。它掃過專案、大方向沒問題——框架、目錄它讀得出來。但那些我昨天只是「講」給它聽、沒寫進任何檔案的約定：哪幾個檔案不准動、某個地方為什麼要這樣寫、之後打算怎麼換，它一概不知道，於是又按最常見的預設猜了一輪。我心裡嘆了口氣，把昨天那串交代再打一次。它照樣可以做得不錯，但我做的事情，其實是「重新哄它一次」。

老實說，我一開始以為問題出在「prompt 下得不夠好」。所以去看了很多下 prompt 的技巧，收藏了一堆「超強咒語」。它們確實有用，但只有那一次有用。下次換個任務、換個對話，我又回到原點，重新拼湊那串話。後來我才發現：真正缺的，不是一句多完整的 prompt，而是**一套不用每次重講、結果可以複製的運作方式**。

## 問題不是「不夠聰明」，是「不可複製」

現在的模型已經很強了。它會 thinking、會查資料、會呼叫工具、會分步驟推理。單看每一次的表現，它常常超出我的預期。

但「單次表現好」跟「能穩定依賴它」是兩回事。開發是一件重複的事：同一種問題會反覆出現，同一套慣例要一直遵守，同一類任務下週還會再來一次。如果每一次都得靠我當下的心情、當下記得多少、當下 prompt 下得多完整，那我得到的就是一個運氣成分很高的協作者，而不是一個工具。

我要的其實很簡單：同一種情境，下一次能得到差不多水準的結果，而且我不必從零交代。這在軟體工程裡有個老名字，叫**可複製性**。我們對建置流程、對測試、對部署，早就要求可複製了，卻在跟 AI 協作這件事上，退回到「每次手動、每次憑感覺」。

## 換一個問法：什麼情境，該用哪個 pattern

所以這個系列想把問題換一個問法。

不是「怎麼把 prompt 寫得更漂亮」，而是：**我在開發時遇到這件事，這時候該拿出哪個 pattern，用 Claude Code 怎麼落地？** 這裡的 pattern，跟我們開發時的 design pattern 一樣，指的是一種可以固定下來、下次直接套用的運作模式，只是我們這裡 pattern 的對象是 AI 不是程式碼。它可能是一份寫好的規則、一個封裝好的 skills、一道自動的 rules、或是幾個 sub-agent 組合起來的流水線。重點是：一旦你把某個情境對應到某個 pattern，它就從「每次重想」變成「每次拿來用」。

這個系列不是 pattern 百科，不會從第一個 pattern 按字母排到最後一個。它是情境先行的：每一天，你會帶著一個具體的、開發時真的會遇到的問題進來——AI 沒讀檔就開始瞎猜、大重構散落在十幾個檔案、上線後湧進一堆雜亂的 feedback——然後我們一起回答「這個場景該用哪個模式，在 Claude Code 裡怎麼落地」。至於那些通用的 pattern 名稱（像 ReAct、Supervisor、Map-Reduce 這些），我會放在每篇的括號裡當對照詞彙，方便之後查到正式術語，但它們不是主角，遇到的情境才是。

## 一個會真的長大的 side project

要讓「情境 → pattern」不流於舉例，最好的辦法是全程用同一個專案。

所以整個系列會綁在同一個 side project 上：一個「稍後閱讀加上 AI 週報摘要」的小型 web 工具。它的功能講起來很單純——使用者把想看的文章連結存進來，agent 幫忙抓內容、產出單篇摘要，到了週末再把這一週存的東西彙整成一份週報。

我選這個題目，單純只是因為它會把三種階段的問題全踩過一遍：

- 一開始它是個一人 MVP。我要獨自把核心功能做出來（存連結、抓內容、單篇摘要），這時候的痛點是「怎麼讓 AI 照我的規矩、想清楚再動手」。
- 接著功能變多，我一個人顧不來每個環節，開始需要把工作拆給不同分身、加上審查，讓交付變成一條可靠的產線。
- 最後它上線了、有了真的使用者。這時候要面對的是會失敗的外部呼叫、爆掉的 context、湧進來的回饋，以及「下一個功能該做什麼」。

換句話說，這個專案在 30 天裡遇到的每一件麻煩事，剛好對應到後面每一天要談的 pattern。如果你手上剛好有想做的真實專案，把名字換掉就好，底下這三個階段的結構是一樣的。

## 這 30 天的地圖

整個系列分成三個 Part，對應「打造並套用你自己的一套 agent 運作模式」的三個層次。

第一個是**建立**（Part 1）。這一段在做地基：用 Claude Code 的長期記憶、規則、技能、守門這幾個機制，把「讓 agent 記得慣例、照著規矩、危險動作被擋下來」固定下來；再加上幾種推理模式，讓它在動手前先想清楚。這一段結束時，你手上會有一組可複製的環境設定，而不是每次重講的口頭交代。

第二個是**組裝**（Part 2）。當單一 agent 顧不來、任務開始散開時，我們用 sub-agent 把幾個角色組合起來——有人拆派、有人執行、有人專門挑錯、有人負責選型。這一段是把個別能力組裝成處理特定任務的產線。

第三個是**應用**（Part 3）。專案上線之後，重點從「把功能做出來」轉到「讓它穩定活著並持續長大」：處理會失敗的呼叫、把爆掉的 context 收好、把湧進來的使用者回饋彙整分類排優先、發想下一批功能，最後把前面學到的東西串成你自己的一條私有 workflow。

三個 Part 之間是遞進的。地基沒鋪好就談組裝，會發現分身之間各說各話；沒有可靠的產線就談上線維運，會被回饋淹沒。所以我們一天一天來，跟著那個 side project 的成長順序走。

## 接下來 30 天，你可以怎麼跟著做

這個系列可以只當讀物，但如果你願意動手，收穫會差很多。

我的建議是：開一個你自己的 side project，可以就是那個稍後閱讀工具，也可以換成任何你真正想做的東西。每天讀完一篇，就把當天那個 pattern 套進你的專案，落地成一份可複製的設定——一份 rule、一個 skill、或一組 sub-agent。三十天下來，你看過了數個 pattern，應該也會有想法去攢出一套屬於你自己、下次能直接拿來用的運作模式。

明天 Day 02，我們從地基的第一塊開始，處理最惱人的那件事：AI 每開一次新對話就忘記專案慣例、要你重講一遍。我們會把該記住的東西寫進 CLAUDE.md 與 auto memory，讓 agent 一啟動就對得上你的專案，不必再從零哄它一次。
