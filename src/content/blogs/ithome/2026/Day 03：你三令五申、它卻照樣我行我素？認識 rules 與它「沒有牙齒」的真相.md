## 昨天寫進了記憶，今天它還是不聽

Day 02 我們把專案慣例寫進了 CLAUDE.md 與 auto memory，讓 agent 一開場就對得上框架／套件、目錄邊界。那一步解決的是「每開新對話就失憶、要你重講」的問題。

但很快你會撞到下一道牆：**它記得了，不代表它照做。**

我在「稍後閱讀 + AI 週報摘要」這個 side project 裡就吃了一次悶虧。我很早就在專案裡立了兩條規矩，白紙黑字寫在規則檔裡：一是「動任何功能前先寫測試」，二是「`db/migrations/` 是生出來的產物，不准手改」。這兩條我覺得夠清楚了，清楚到不像會被誤解。

結果那天我請 agent 幫我在摘要模組加一個「同一篇文章只留一份」的去重邏輯。它很有效率，直接改了三個檔案、把功能寫好、順手還「幫我」動了一支 migration 檔把 schema 調成它想要的樣子。測試？一行都沒有。整段流程它跑得理直氣壯，完全沒有停下來問一句。

我第一個反應是：規則明明就在那裡，它是沒讀到嗎？翻了對話紀錄才發現，它讀到了，甚至在回覆裡還複述了一遍「依照專案慣例應先撰寫測試」。讀到了、也複述了，然後照樣我行我素。

這不是它壞掉了。這是 rules 這層本來就是這樣運作的。

## rules 在哪一層：會讀，但不保證遵守

要理解剛剛那件事，得先把 Claude Code 的機制分成三層來看。這是理解整個 Part 1 的地基，這裡先講清楚：

| 層級 | 代表機制 | 對 AI 的作用 |
|---|---|---|
| context（脈絡） | CLAUDE.md、auto memory、rules | 進入模型的視野，會被讀到、參考 |
| invoke（呼叫） | skills、subagents、工具 | 觸發才載入的能力與分身 |
| enforce（強制） | hooks、permissions | 真正能攔下動作、有否決權 |

rules 屬於最上面那層。它的本質是**行為準則**，是你寫給 agent 看的「我希望你這樣做事」。它會被放進 context，模型在推理時會參考它，多數時候也會照著走。但它終究是「參考資料」，不是「執行時的閘門」。模型有沒有遵守，取決於它當下對任務的判斷、對規則的權重分配，以及有沒有被別的目標蓋過去。

換句話說，rules 是說服，不是強制。它像公司牆上貼的守則，寫得再工整，也擋不住一個當下覺得「這次情況特殊」的人。差別只在於，AI 不會覺得愧疚而已。

所以那天的去重任務，agent 心裡（如果能這樣講）真正的目標是「把去重功能做出來」。寫測試、別碰 migration，這兩條規則在它眼中是背景裡的偏好，一旦跟主目標的效率有衝突，它就自己權衡掉了。這不是它不尊重你，是這層機制天生就沒有否決權。

想通這件事，你對 rules 的期待就會校準：**它負責定義「對的做法長什麼樣」，但它擋不下「錯的做法」。** 真的要擋，得靠最底下那層 enforce——那是 Day 05 的主題。

## 什麼該放 rules，什麼別只靠它

分清楚層級之後，判斷就簡單了。用一個問題篩：**這條規矩被違反時，後果是「風格不一致、要重寫」，還是「造成無法回復的損害」？**

前者適合放 rules。命名慣例、測試風格、目錄結構偏好、commit 訊息格式、要用哪個套件不用哪個——這類東西就算 agent 偶爾漏掉，你 review 時看得到，改回來成本也低。rules 在這裡的價值很高：它讓 agent 大部分時候都走在對的路上，省掉你反覆糾正的力氣。

後者不能只靠 rules。刪除檔案、推上 production、改動 migration、寫入某個絕對不能碰的目錄——這些一旦發生就來不及了，你不能把防線押在「希望它這次有讀進去」。這類必須交給 enforce 層用 hook 或 permission 真正擋下。

我那次的教訓正好橫跨這兩類。「先寫測試」漏了，頂多是我發現後補上、或退回重來，屬於可接受的 rules 範疇。但「手改 migration」造成的是 schema 漂移，這種就不該只寫在 rules 裡期待它自律——它本來就該是 enforce 層的守備範圍，我當時把不對的東西託付給了不對的層。

所以判斷不是「rules 好不好用」，而是「這件事配不配只用 rules」。把它當唯一防線的地方，多半是你把 enforce 該做的事錯放上來了。

## 怎麼把 rules 寫得真的有用

既然 rules 靠的是說服，那它的措辭就直接決定被遵守的機率。同樣一件事，寫法不同，效果差很多。我自己收斂出三個原則。

第一，**具體、可檢查，別寫價值觀。** 「請寫出高品質的程式碼」這種話等於沒寫，模型無從判斷有沒有做到。改成可驗證的描述：「每個新增的 public function 都要有對應的測試檔，放在 `tests/` 下同名目錄」。可檢查的規則，模型比較知道怎麼對照、你事後也驗得了。

第二，**講清楚「為什麼」，不只是「不准」。** 「不要手改 `db/migrations/`」不如寫「`db/migrations/` 由 migration 工具生成，手改會造成 schema 與工具狀態不一致，要改 schema 請改 `schema.prisma` 再重新生成」。給了原因與替代路徑，模型比較不會在覺得「特殊情況」時自作主張，因為你已經先把它想繞過的理由堵掉了。

第三，**聚焦，別把規則寫成長篇大論。** 規則檔越長，每一條被稀釋得越嚴重——這是 context 的現實，塞太多等於沒有重點。把最關鍵、最常被違反的幾條擺前面、講明白，勝過洋洋灑灑列三十條讓模型自己抓重點。

## 用 Claude Code 落地

在 Claude Code 裡，專案規則放在 `.claude/rules/` 底下，用 markdown 寫。針對這個 side project，我會這樣寫一份開發慣例的規則檔：

```markdown
# 開發慣例

適用範圍：本專案（稍後閱讀 + AI 週報摘要）所有程式碼異動。

## 測試

- 每個新增或修改的功能，MUST 先寫測試再寫實作。
- 測試檔放在 tests/ 下，與被測模組同名（例：summarizer.ts → tests/summarizer.test.ts）。
- 用 bun test 執行；提交前所有測試 MUST 通過。

## 命名

- 檔名一律 kebab-case；元件檔用 PascalCase。
- 非同步函式以動詞開頭（fetchArticle、buildWeeklyReport）。

## 禁改區

- db/migrations/ 由 migration 工具生成，NEVER 手改。
  要調整 schema 請改 schema.prisma 再重新生成。
- 這條若被違反會造成 schema 與工具狀態不一致，屬於高風險。
```

注意這份檔案裡我刻意標了一句「屬於高風險」。這不是給模型看的裝飾，是給我自己看的提醒：**凡是標了高風險的條目，代表它不該只活在 rules 裡，Day 05 我會為它補上 hook。** rules 檔本身可以順便當成一份「哪些事還需要 enforce 補強」的清單。

寫好之後，你可以直接請 agent 做一件會踩到規則的事，看它怎麼反應——多數時候它會照做，但你會很快找到那幾條「講了也未必聽」的縫。找到縫，就是 Day 05 的工作起點。

## 對照概念：程序記憶與行為約束

把視野拉高一點，rules 對應的是 agent 領域裡兩個概念的交會。

一個是**程序記憶（procedural memory）**。相對於「記得專案是什麼」的事實性記憶（那比較接近 Day 02 的 CLAUDE.md），程序記憶記的是「事情該怎麼做」——工作流程、慣例、步驟。rules 就是把你的開發程序寫成模型每次都讀得到的常駐知識，讓它不必每次重新摸索你的做事方式。

另一個是**行為約束與護欄（guardrails）**。護欄的完整光譜從「軟性引導」一路到「硬性攔截」都有，而 rules 明確落在最軟的那端：它塑形行為，但不保證合規。真正有牙齒的護欄——會逕自否決動作的那種——在架構上是另一種東西，得靠執行期的攔截機制。這也是為什麼一套完整的安全設計不會只有 rules，它會是「軟引導 + 硬攔截」的組合。

把這兩個概念疊在一起看，你就會明白 rules 的定位：它是**寫給模型的程序記憶，扮演最軟的那層護欄**。它很有用，但你要用在它擅長的地方，別叫一份備忘錄去當保全。

## 今日實踐任務

今天替你的 side project 建一份開發慣例規則，並親手驗證它的極限。

1. 在專案的 `.claude/rules/` 下建一份規則檔，至少涵蓋三類：命名慣例、測試要求、一個禁改區（例如 migration 目錄或某個生成產物目錄）。
2. 每一條都用上前面三個原則寫：具體可檢查、附上原因與替代路徑、只留最關鍵的幾條。
3. 刻意測它一次：開新對話，請 agent 做一件會同時踩到「測試要求」和「禁改區」的任務，觀察它的實際行為。
4. 把觀察記下來：哪幾條被遵守了、哪幾條被繞過了。被繞過的那些，就是明天要交給 enforce 層補強的候選名單。

驗收標準：

- 規則檔存在於 `.claude/rules/` 下，三類慣例齊全，每條都是可檢查的描述而非空泛口號。
- 你能明確說出「這次測試裡，agent 遵守了哪些、繞過了哪些」——重點不是它全部照做，而是你親眼確認了 rules「沒有牙齒」的地方在哪。
- 高風險的禁改區條目已被標記出來，列入待補強清單。

如果你的測試結果是 agent 全部乖乖照做，別急著開心，多試幾種任務、幾種措辭，那道縫一定找得到。找到它，你才真正理解了這層機制的邊界。

明天 Day 04，我們換個問題：有些複雜任務你每次都要對 agent 重講一長串前因後果，講到自己都煩。與其反覆口述，不如把它封裝成 skills 與 subagents——需要時觸發才載入，把重複的複雜流程變成一個叫得動的能力。
