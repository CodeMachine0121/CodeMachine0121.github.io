## 從一段怎麼看都不順眼的 code 說起

前兩天我們把兩種推理模式練了一遍。Day 07 的 CoT 是純推理，逼 agent 把腦子裡的推導攤開再下 code；Day 08 的 ReAct 則是推理加行動，一條路邊走邊修，每一步都踩在上一個觀察上。這兩者有個共通點：它們都假設「有一條路可以走」，差別只在走的時候要不要出去查證。

但在這個 Go 後端裡，你很可能會遇到一個不太一樣的問題。Day 08 那個抓取模組（`internal/fetcher`）裡有一個函式，負責把抓回來的 HTML 清成乾淨正文——去掉導覽列、廣告、頁尾，只留文章本體。它能跑，`go test` 也綠，但長得很難看——巢狀了四層 `if`，還混了兩種解析方式，過幾天回來看都要重讀一遍才懂。想重構，卻卡在一個很煩的點：這段 code 不是只有一種救法。

大概想得到三條路。可以就地清理，把巢狀條件拆成幾個具名的小函式，動靜最小；可以換一套做法，改用現成的正文抽取套件（例如 go-readability）取代手刻的節點篩選，改動中等但邏輯會清爽很多；也可以乾脆把它拆成一條 pipeline，去導覽、去廣告、去頁尾、正規化各自一個純函式串起來，最漂亮但改動最大、還要動到 `internal/fetcher` 的呼叫端。三條路各有代價，一時之間真的不知道選哪個。

這種時候你要是直接叫 agent「幫我重構這段」，它會很乾脆地挑一條路走到底——通常是它第一個想到的那條。走完你才發現，欸，另外兩條說不定更適合，但這時候已經改了一堆，要退回來重比很麻煩。問題不在 agent 不會重構，而在它沒有「先把幾條路攤開來比一比」這個動作。今天要建的，就是這個動作背後的迴圈。

## 判斷：什麼時候值得展開成一棵樹

先講清楚，這個做法不是每題都該拿出來，它其實偏重。要判斷值不值得，可以看三個訊號。

第一，**這題有沒有好幾種站得住腳的解法**。如果答案基本上是唯一的，或者幾條路差異小到怎麼選都差不多，那展開多路徑純粹是浪費 token。像「把這個變數改個名字」這種，一條路就到底了，用 CoT 想清楚甚至直答就好。

第二，**錯誤的成本高不高**。上面那個重構之所以值得慎重，是因為三條路的改動幅度差很多，選錯了要嘛留下技術債、要嘛白改一輪。反過來說，如果錯誤只要五分鐘就能改回來，那不如直接挑一條試，錯了再說——那正是 ReAct 邊走邊修的場合。

第三，**搜索空間夠不夠大、需不需要試錯比較**。有些問題你事先根本不知道哪條路會通，得先展開幾個候選、各自往下推一小段、看哪個苗頭對，再決定要不要繼續。演算法選型、資料結構設計、「這個 bug 有好幾個可能成因」的排查，都屬於這一類。

三個訊號可以濃縮成一句判斷：**多方案、需要比、而且選錯不便宜，才值得展開成樹。** 只要有一個不成立，多半用前兩天的 CoT 或 ReAct 就夠了。一開始你很容易對什麼都想展開多路徑，覺得這樣比較「周全」，但吃過幾次苦頭就會學著克制——大部分日常任務根本沒有那麼多條路，硬展開只是拖慢自己、燒掉額度。

## 建立：把內迴圈攤成一棵可回溯的樹

想清楚該用之後，這個做法的骨架其實只有四個動作，你可以把它想成在一棵樹上做事。

**展開（branch）**：對同一個問題，一次生出 N 條互不相同的候選路徑。重點在「互不相同」——如果三條路本質一樣、只是措辭不同，那就白展開了。所以下指令時要明確要求方案之間在策略上要有真正的差異。

**評分（evaluate）**：讓每條路徑各自對著同一組標準打分，而不是只憑感覺說「這個比較好」。標準要事先講明，例如改動幅度、可讀性、對現有 `go test` 的衝擊、未來擴充的容易度。每條路都用同一把尺量，才比得出高下。

**選優（select）**：根據評分挑出最有希望的一條或幾條往下走。這裡可以只留一條，也可以留前兩名繼續深入——留幾條取決於你願意花多少成本。

**回溯（backtrack）**：關鍵動作，也是這個做法跟「一次生三個方案讓你選」最大的差別。當某條路往下推之後發現它走不通（例如深入才發現會破壞 `internal/fetcher` 一個原本沒注意到的呼叫端），要能把這條劣枝剪掉、退回上一個分岔點，改走次優的那條，而不是明知是死路還繼續往下走。

把這四個動作連起來，就是「展開幾條 → 各自評分 → 選優往下 → 走不通就回溯換一條」。換個角度看，這其實是把前兩天那個線性的內迴圈攤成一棵**有分支、可剪枝回溯的樹**——不再是一條路想到底，而是在解法空間上做一次有評估的小搜索。CoT 是一條線性的推導，這裡是一棵樹，需要的規劃味也更重。（這棵樹狀推理在文獻裡叫 Tree-of-Thoughts，名字記著方便查資料就好，重點是那個「展開、評估、剪枝、回溯」的動作。）

## 用 Claude Code 落地

四個動作想清楚了，接下來是把它固定成能重複用的東西。主流的做法是封裝成 skill——Day 04 談過，重複的複雜流程該封裝、觸發才載入。展開多路徑評估正是這種流程：每次都要交代「展開幾條、要真正不同、用哪組標準評分、走不通要回溯」，與其每次重打，不如寫成一個 skill，之後遇到選型或重構直接觸發。

skill 的骨架跟 Day 04 一樣，`.claude/skills/` 底下開一個資料夾，放一份 `SKILL.md`；它的 `description` 就是觸發條件，四個動作和一份評分表範本寫進步驟裡：

```
.claude/skills/
└── explore-options/
    └── SKILL.md
```

```
---
name: explore-options
description: 當一個重構或技術選型有多種站得住腳的方案、且選錯成本高（會留下技術債或白改一輪）時使用。展開多條策略上不同的路徑，各自對同一組標準評分，選優，必要時回溯換路。
---

## 什麼時候用

多方案、需要比、而且選錯不便宜時才展開；答案唯一、或改錯五分鐘就能還原的，別用這個 skill，直接做。

## 流程

1. 展開三條「策略上真正不同」的路徑，各自簡述做法；若本質相同、只是措辭不同，重展開。
2. 用同一組標準替每條打分（1-5），標準先講明：改動幅度（越小越高）、可讀性、對現有測試的衝擊（越小越高）、未來擴充容易度。任務不同可換一組標準，但每條路必須用同一把尺量。
3. 列出評分表，選總分最高的一條，說明為何選它、為何淘汰其餘。
4. 若選中的那條深入後發現致命問題（例如破壞某個原本沒注意到的呼叫端），明講、回溯，改採次高分的路徑，更新結論。
5. 只針對選定路徑輸出方案，先不要動手改其他候選。
```

寫好之後，任何對話你只要說「用 explore-options 評估 internal/fetcher 那個 HTML 清理函式該怎麼重構」，它就照這套展開，不用再口述一遍。這正是 Day 04 那個判斷的延伸：固定流程、想在當前對話裡完成，就封裝成 skill。

如果只是一次性、不想特地建 skill，也可以把同一套直接寫成一段 prompt，請 agent 在一次回應裡走完。重點跟 skill 一樣：把「要真正不同」和「用同一組標準評分」講死，不然它會給你三個換湯不換藥的方案、再含糊地說第一個最好。

```
針對 internal/fetcher 裡那個把抓回來的 HTML 清成正文的函式（目前巢狀四層 if、混了兩種解析方式，能跑、go test 也綠，但很難讀），請用展開多路徑評估的方式重構，不要直接動手改：

1. 展開三條「策略上真正不同」的重構路徑，簡述各自怎麼做：
   - 就地清理（把巢狀 if 拆成具名小函式）
   - 改用現成正文抽取套件（如 go-readability）取代手刻節點篩選
   - 拆成 pipeline（去導覽／去廣告／去頁尾／正規化各一個純函式串接）
2. 用同一組標準替每條路徑打分（1-5）：
   - 改動幅度（越小越高分）
   - 可讀性提升
   - 對現有 go test 的衝擊（越小越高分）
   - 未來擴充容易度
3. 列出評分表，選出總分最高的一條，說明為何選它、為何淘汰其餘兩條。
4. 如果選中的那條在深入時發現有致命問題（例如破壞 internal/fetcher 某個呼叫端），
   請明講、回溯，改採次高分的路徑，並更新結論。

最後只針對選定的路徑輸出重構方案（含要新增／修改的 Go 函式簽章），先不要改其他候選，也先不要跑 go test。
```

遇到重要決策、想讓每條路徑獨立展開互不影響，可以再往上一級，用 subagent 分頭評估。派三個 subagent 各自負責一條路徑，每個都拿到同一份 Go 程式碼和同一組評分標準，各自把方案推深、給出評分與風險；主 agent 收回三份結果，對齊評分、選優，必要時要求得分最高的那個再往下走一段、確認它動到的呼叫端都還能 `go test` 通過。好處是每條路徑獨立展開，不會像單次回應那樣互相污染（agent 常會下意識地讓後面的方案去「襯托」它偏好的第一個）。從迴圈的角度看，這一步等於把「逐條試」的內迴圈並行化了——本來要一條一條展開評估，攤成三個 subagent 同時各跑一支，是把迴圈平行化的雛形。

至於 rule，適合放的不是整套協議，而是一句觸發條件。Day 03 講過 rule 沒有強制力，加上這種展開很貴、不該一直掛著跑，所以 rule 只負責提醒習慣、把活交給 skill：例如「遇到重構或選型且方案不只一種時，先用 explore-options 展開多路徑再動手」，讓這個習慣固定下來。

要提醒的是，不管哪種做法都比 CoT、ReAct 貴——你等於把一題算了好幾遍。所以回到前面的判斷：只在多方案、選錯不便宜的題目上動用它。

## 對照概念：帶分支與剪枝的樹狀推理

如果把這幾天的推理模式擺在一起看，脈絡會很清楚。線性的推理是一條鏈，從頭想到尾；加上行動與觀察，就成了邊走邊查的迴圈。而今天談的，是把那條內迴圈攤成一棵樹：從同一個起點分出好幾條枝，每條枝各自往下探、各自評估，好的留、壞的剪，走不通就退回分岔點換一條。

這對應到業界講的「探索與分支決策」那一類技巧——核心精神是把解法空間當成一個可以搜索的樹，用評估函式去引導要往哪條枝深入、哪條枝該淘汰。它跟純粹的線性推理最大的不同，就在那個「回溯」的能力：線性推理一旦走偏，只能一路錯到底；樹狀推理則允許承認某條路是死路、退回去換一條。代價是明顯更高的運算成本，所以它天生就是為「搜索空間大、需要試錯比較」的問題準備的，不是通用的萬用解。

## 今日實踐任務

繼續推進這個稍後閱讀後端。今天不改重構那段 Go code，改處理一個更貼近產品的決策：**單篇文章該用哪一種摘要策略。**

同一篇文章，摘要其實有好幾種寫法。你可以做「重點條列式」（抽三到五個要點），可以做「一段話式」（濃縮成一小段連貫敘述），也可以做「問答式」（把文章拆成幾個它回答了的問題）。這三種各有適合的文章類型，一時也難說哪個最好——正好是展開成樹的好題目。

請照這個流程做一次：

1. 挑一篇你實際存過的文章（技術長文、新聞、教學都可以）。
2. 讓 agent 展開這三種摘要策略，各自對同一篇文章產出一版摘要。
3. 用同一組標準替三版打分：資訊完整度、可掃讀性、對「一週後回頭掃週報」這個使用情境的貼合度。
4. 選出最適合這篇的一種，說明為何選它、為何淘汰其餘兩種。
5. 刻意做一次回溯練習：假設選中的那版被發現「漏掉了文章最重要的一個結論」，要求 agent 承認、剪掉這條，改採次優策略並重新輸出。

驗收標準：

- 三版摘要在策略上真的不同，不是同一種寫法換句話說。
- 有一張用同一組標準打分的評分表，不是空泛地說「第一個比較好」。
- 選優有明確理由，且說得出為什麼淘汰另外兩個。
- 回溯那一步能真的退回、換路，而不是硬凹原本的選擇。

做完你會發現，這個過程留下的評分表本身就有價值——下次遇到同類型的文章，你已經知道該偏哪種策略，不用每篇都重展開一次。這正是這套做法划算的地方：貴，但貴在幫你把一個會反覆出現的決策一次想清楚。

## 明天 Day 10

推理三連到這裡告一段落——線性推、邊走邊查、展開成樹，各有各的場合。但這三種都還是在「想清楚一件事」。明天要換個方向：當你要一次生出一個結構化的大東西，比如一整組 API endpoints（links／tags／report 各自的 handler、service、query 層），最怕的不是某一支寫不出來，而是各層之間簽章對不上、命名不一致——service 叫 `CreateLink`、handler 卻叫 `AddLink`，query 又是另一套。這時候與其線性地一路寫下去，不如先把整組端點與各層函式簽章的骨架立好、對齊，再平行地把每一層的實作填進去。明天我們談 Skeleton-of-Thought，先搭骨架再填肉，讓大結構從一開始就穩住。
