## 從一段怎麼看都不順眼的 code 說起

前兩天我們把兩種推理模式練了一遍。Day 07 的 CoT 是純推理，逼 agent 把腦子裡的推導攤開再下 code；Day 08 的 ReAct 則是推理加行動，一條路邊走邊修，每一步都踩在上一個觀察上。這兩者有個共通點：它們都假設「有一條路可以走」，差別只在走的時候要不要出去查證。

但昨天晚上我在稍後閱讀工具裡撞到一個不太一樣的問題。摘要模組裡有一個函式，負責把抓回來的文章去重、切段、再排出重點順序。它能跑，但長得很難看——巢狀了四層迴圈，還混了兩種資料結構，我自己過幾天回來看都要重讀一遍才懂。想重構，卻卡在一個很煩的點：這段 code 不是只有一種救法。

我大概想得到三條路。可以就地清理，把巢狀迴圈拆成幾個具名的小步驟，動靜最小；可以換一套資料結構，用一個中介的索引表把去重和排序解耦，改動中等但邏輯會清爽很多；也可以乾脆把它拆成一條 pipeline，每個階段一個純函式，最漂亮但改動最大、還要動到呼叫端。三條路各有代價，我一時之間真的不知道選哪個。

這種時候你要是直接叫 agent「幫我重構這段」，它會很乾脆地挑一條路走到底——通常是它第一個想到的那條。走完你才發現，欸，另外兩條說不定更適合，但這時候已經改了一堆，要退回來重比很麻煩。問題不在 agent 不會重構，而在它沒有「先把幾條路攤開來比一比」這個動作。這就是今天要談的 Tree-of-Thoughts。

## 判斷：什麼時候值得展開成一棵樹

先講清楚，這個模式不是每題都該拿出來，它其實偏重。要判斷值不值得，我會看三個訊號。

第一，**這題有沒有好幾種站得住腳的解法**。如果答案基本上是唯一的，或者幾條路差異小到怎麼選都差不多，那展開多路徑純粹是浪費 token。像「把這個變數改個名字」這種，一條路就到底了，用 CoT 想清楚甚至直答就好。

第二，**選錯的成本高不高**。上面那個重構之所以值得慎重，是因為三條路的改動幅度差很多，選錯要嘛留下技術債、要嘛白改一輪。反過來說，如果選錯只要五分鐘就能改回來，那不如直接挑一條試，錯了再說——那正是 ReAct 邊走邊修的場合。

第三，**搜索空間夠不夠大、需不需要試錯比較**。有些問題你事先根本不知道哪條路會通，得先展開幾個候選、各自往下推一小段、看哪個苗頭對，再決定要不要繼續。演算法選型、資料結構設計、「這個 bug 有好幾個可能成因」的排查，都屬於這一類。

三個訊號可以濃縮成一句判斷：**多方案、需要比、而且選錯不便宜，才值得展開成樹。** 只要有一個不成立，多半用前兩天的 CoT 或 ReAct 就夠了。老實說，我一開始很容易對什麼都想展開多路徑，覺得這樣比較「周全」，後來吃了幾次苦頭才學會克制——大部分日常任務根本沒有那麼多條路，硬展開只是拖慢自己、燒掉額度。

## 建立：展開、評分、選優、回溯

想清楚該用之後，這個模式的骨架其實只有四個動作，你可以把它想成在一棵樹上做事。

**展開（branch）**：對同一個問題，一次生出 N 條互不相同的候選路徑。重點在「互不相同」——如果三條路本質一樣、只是措辭不同，那就白展開了。所以下指令時要明確要求方案之間在策略上要有真正的差異。

**評分（evaluate）**：讓每條路徑各自對著同一組標準打分，而不是只憑感覺說「這個比較好」。標準要事先講明，例如改動幅度、可讀性、對現有測試的衝擊、未來擴充的容易度。每條路都用同一把尺量，才比得出高下。

**選優（select）**：根據評分挑出最有希望的一條或幾條往下走。這裡可以只留一條，也可以留前兩名繼續深入——留幾條取決於你願意花多少成本。

**回溯（backtrack）**：關鍵動作，也是這個模式跟「一次生三個方案讓你選」最大的差別。當某條路往下推之後發現它撞牆了（例如深入才發現會破壞一個原本沒注意到的呼叫端），要能把這條劣枝剪掉、退回上一個分岔點，改走次優的那條，而不是硬著頭皮在死路上繼續。

把這四個動作連起來，就是「展開幾條 → 各自評分 → 選優往下 → 走不通就回溯換一條」。它跟 CoT 的差別在於：CoT 是一條線性的推導，ToT 是一棵有分支、可以剪枝的樹。它需要的規劃味也更重——你其實是在讓 agent 對解法空間做一次有評估、有淘汰的小型搜索。

## 用 Claude Code 落地

落地有兩種做法，看你要多正式。

輕量的做法，是把這四個動作寫成一段 prompt，直接請 agent 在一次回應裡走完。重點是把「要真正不同」和「用同一組標準評分」講死，不然它會給你三個換湯不換藥的方案、再含糊地說第一個最好。

```
針對這段函式的重構，請用 Tree-of-Thoughts 的方式進行，不要直接動手改：

1. 展開三條「策略上真正不同」的重構路徑，簡述各自怎麼做。
2. 用同一組標準替每條路徑打分（1-5）：
   - 改動幅度（越小越高分）
   - 可讀性提升
   - 對現有測試的衝擊（越小越高分）
   - 未來擴充容易度
3. 列出評分表，選出總分最高的一條，說明為何選它、為何淘汰其餘兩條。
4. 如果選中的那條在深入時發現有致命問題（例如破壞某個呼叫端），
   請明講、回溯，改採次高分的路徑，並更新結論。

最後只針對選定的路徑輸出重構方案，先不要改其他候選。
```

比較正式、也比較適合重要決策的做法，是用 subagent 分頭評估。你派三個 subagent 各自負責一條路徑，每個都拿到同一份程式碼和同一組評分標準，各自把方案推深、給出評分與風險；主 agent 收回三份結果，對齊評分、選優，必要時要求得分最高的那個再往下走一段確認沒撞牆。這樣做的好處是每條路徑是獨立展開的，不會像單次回應那樣互相污染（agent 常會下意識地讓後面的方案去「襯托」它偏好的第一個）。你也可以在 rule 裡替這種任務留一句觸發條件，例如「遇到重構或選型且方案不只一種時，先展開多路徑評估再動手」，讓這個習慣固定下來。

要提醒的是，不管哪種做法都比 CoT、ReAct 貴——你等於把一題算了好幾遍。所以回到前面的判斷：只在多方案、選錯不便宜的題目上動用它。

## 對照概念：帶分支與剪枝的樹狀推理

如果把這幾天的推理模式擺在一起看，脈絡會很清楚。線性的推理是一條鏈，從頭想到尾；加上行動與觀察，就成了邊走邊查的迴圈。而今天談的，是把推理攤成一棵樹：從同一個起點分出好幾條枝，每條枝各自往下探、各自評估，好的留、壞的剪，走不通就退回分岔點換一條。

這對應到業界講的「探索與分支決策」那一類技巧——核心精神是把解法空間當成一個可以搜索的樹，用評估函式去引導要往哪條枝深入、哪條枝該淘汰。它跟純粹的線性推理最大的不同，就在那個「回溯」的能力：線性推理一旦走偏，只能將錯就錯到底；樹狀推理則允許承認某條路是死路、退回去換一條。代價是明顯更高的運算成本，所以它天生就是為「搜索空間大、需要試錯比較」的問題準備的，不是通用的萬用解。

## 今日實踐任務

繼續推進稍後閱讀工具。今天不改重構那段 code，改處理一個更貼近產品的決策：**單篇文章該用哪一種摘要策略。**

同一篇文章，摘要其實有好幾種寫法。你可以做「重點條列式」（抽三到五個要點），可以做「一段話式」（濃縮成一小段連貫敘述），也可以做「問答式」（把文章拆成幾個它回答了的問題）。這三種各有適合的文章類型，一時也難說哪個最好——正好是展開成樹的好題目。

請照這個流程做一次：

1. 挑一篇你實際存過的文章（技術長文、新聞、教學都可以）。
2. 讓 agent 用 ToT 展開這三種摘要策略，各自對同一篇文章產出一版摘要。
3. 用同一組標準替三版打分：資訊完整度、可掃讀性、對「一週後回頭掃週報」這個使用情境的貼合度。
4. 選出最適合這篇的一種，說明為何選它、為何淘汰其餘兩種。
5. 刻意做一次回溯練習：假設選中的那版被發現「漏掉了文章最重要的一個結論」，要求 agent 承認、剪掉這條，改採次優策略並重新輸出。

驗收標準：

- 三版摘要在策略上真的不同，不是同一種寫法換句話說。
- 有一張用同一組標準打分的評分表，不是空泛地說「第一個比較好」。
- 選優有明確理由，且說得出為什麼淘汰另外兩個。
- 回溯那一步能真的退回、換路，而不是硬凹原本的選擇。

做完你會發現，這個過程留下的評分表本身就有價值——下次遇到同類型的文章，你已經知道該偏哪種策略，不用每篇都重展開一次。這正是 ToT 划算的地方：貴，但貴在幫你把一個會反覆出現的決策一次想清楚。

推理三連到這裡告一段落——線性推、邊走邊查、展開成樹，各有各的場合。但這三種都還是在「想清楚一件事」。明天要換個方向：當你要一次生出一個結構化的大東西，比如多頁的 UI、一份很長的文件，最怕的是越寫越歪、寫到後面跟前面對不上。這時候與其線性地一路寫下去，不如先立好整體骨架、再平行地把每一段的肉填進去。明天我們談 Skeleton-of-Thought，先搭骨架再填肉，讓大結構從一開始就穩住。
