# 部落格文章多代理協作撰寫流程

適用範圍：`src/content/blogs/**` 下篇幅較長、需要高品質的文章（尤其是 iThome 系列的 Pattern 文）。當使用者要求「用多個 sub-agent 產出再整合」時，依本流程執行。

此流程的目標：用「多份獨立草稿 + 一次整合」取代「一次寫成」，靠不同視角的差異化素材截長補短，產出比單一視角更扎實、更好讀的定稿。

## 0. 前置：先定位，再開工

在啟動任何 sub-agent 前，**MUST** 先確認該篇在系列中的定位：

- 讀 [`../docs/ithome-2026-series-plan.md`](../docs/ithome-2026-series-plan.md)，取得該篇的**標題、`datetime`、在路線圖中的階段定位、預先設計的 side project**。
- 讀**前一篇**文章，掌握其結尾的「明天 Day N＋1」預告、敘事節奏與版式（新篇要承接）。
- 讀 [`blog-writing-style.md`](./blog-writing-style.md)，確認硬性風格規則。

這些定位資訊要寫進每個 sub-agent 的 prompt，否則各代理產出會發散、無法整合。

## 1. 第一階段：N 個作者各寫一篇「完整」文章

- 預設啟動 **3 個** sub-agent，**MUST 在同一則訊息內並行啟動**（多個 tool use），不要序列等待。
- 每個 agent 產出的是**可獨立成篇的完整文章**（含 frontmatter、所有章節、今日實踐任務），**NEVER** 只寫某個段落分塊——分塊整合會產生拼接感，且無法互相取代擇優。
- 為了讓整合者有差異化素材可取捨，給每個 agent 一個**不同的切入視角（lens）**，例如：
  - **理論深度**：把原理、為何有效、與相鄰概念的本質差異講透。
  - **工程實作**：重心放在可直接複製的設定檔、可操作的範例與驗收。
  - **敘事可讀**：用貼近開發者日常的場景、比喻、故事線串起資訊。
- 每個 agent 的 prompt **MUST** 包含：該篇定位（標題／日期／parent／前後篇銜接）、要讀的參考檔案路徑、完整硬性風格規則、必備內容清單、以及「直接回傳整篇 markdown、不要寫檔」的指示。
- 用 `general-purpose` agent type。產出以回傳文字交回，**此階段不寫檔**。

## 2. 第二階段：1 個整合者合成定稿

- 第一階段全部回來後，再啟動**第 4 個** sub-agent 作整合者。
- 把 N 份草稿全文（或精華摘要 + 關鍵段落原文）一併放進整合者 prompt，並標注「每份最強的部分是什麼」，引導它擇優。
- 整合原則 **MUST** 明確寫入 prompt：
  - 這是「從各篇取最強段落融成一篇」，**NEVER** 只是「選一篇最好的」。
  - 去除跨篇重複，保留最佳表達；全篇語氣、用詞統一，不可有拼接感。
  - 篇幅與前一篇相當或略長即可，**寧精勿濫**；同類素材（如兩段不同的範例 trace）擇一最佳者，不要全放。
- 由整合者用 Write **直接寫入正確檔名**；檔名規則見 [`blog-writing-style.md`](./blog-writing-style.md)（**NEVER** 含 ASCII 斜線）。
- 要求整合者回報：取捨清單（從哪篇取了什麼）、字數概估、風格規則自查清單。

## 3. 第三階段：主流程驗證

整合者寫檔後，主流程 **MUST** 自行驗證實際檔案內容（不要只信回報）：

- frontmatter 五欄齊全（`title`、`datetime`、`description`、`image`、系列篇加 `parent`）。
- 正文從 h2（`##`）起始、**無 h1**；標題無反引號行內程式碼；正文無 emoji。
- 結尾有「明天 Day N＋1」預告。
- **注意**：`grep '^# '` 可能命中**程式碼區塊內**的 `#`（如示範用的設定檔內容），那是允許的例外；命中時要回頭確認該行確實被 ```` ``` ```` 包住，不可誤判為正文 h1。

## 4. 收尾

- 定稿確認後，視情況更新 [`../docs/ithome-2026-series-plan.md`](../docs/ithome-2026-series-plan.md) 的「目前進度」。
